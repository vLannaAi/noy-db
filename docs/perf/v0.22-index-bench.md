# Lazy-mode index benchmark ‚Äî v0.23 acceptance bench

**Acceptance gate (issue [#270](https://github.com/vLannaAi/noy-db/issues/270)):**
p95 < 250ms on three pilot-2 queries at 50K records, cold start, on `to-browser-idb`.

## Where the gate is enforced

Two complementary benches ship:

| Harness | Backend | CI integration | Purpose |
|---|---|---|---|
| `packages/hub/scripts/bench-lazy-indexes.mjs` | `to-memory` | Manual via `node` | Hub-layer baseline ‚Äî measures query dispatch + decrypt cost without any adapter round-trip. |
| `packages/to-browser-idb/__tests__/bench-lazy-indexes.test.ts` | `to-browser-idb` via `fake-indexeddb` | Opt-in via `NOYDB_BENCH=1` | Exercises the real IDB-backed adapter. Asserts the 250ms p95 gate at the smoke scale (1K records) on every PR where the env var is set. |

Both are vitest-driven, so the second one integrates into existing package-level `pnpm test` pipelines with a single env-flag flip.

## Running the bench

```bash
# Smoke gate ‚Äî ~1s, asserts the 250ms p95 gate
NOYDB_BENCH=1 pnpm --filter @noy-db/to-browser-idb test

# Report-only ‚Äî any scale above 1K, emits numbers but skips the gate
# (fake-indexeddb is known slow at sustained IDB traffic, see below)
NOYDB_BENCH=1 NOYDB_BENCH_RECORDS=5000 NOYDB_BENCH_RUNS=10 \
  pnpm --filter @noy-db/to-browser-idb test

# Hub baseline ‚Äî to-memory adapter, no IDB
pnpm --filter @noy-db/hub build
node packages/hub/scripts/bench-lazy-indexes.mjs           # 50K
node packages/hub/scripts/bench-lazy-indexes.mjs 5000      # 5K smoke
```

## Smoke-scale numbers ‚Äî 1K records, `to-browser-idb` + `fake-indexeddb`, Node 20.19.3

| query | p50 (ms) | p95 (ms) | n rows |
|---|---:|---:|---:|
| byClient | 0.01 | 0.10 | 5 |
| byPeriod | 0.14 | 0.35 | 250 |
| byFilingStatus | 0.18 | 0.65 | 50 |

All three queries p95 **< 0.7 ms** ‚Äî comfortably under the 250ms gate.
The smoke run seeds and executes the matrix in ~1.5 seconds total.

## Report-only ‚Äî 5K records, same stack

| query | p50 (ms) | p95 (ms) | n rows |
|---|---:|---:|---:|
| byClient | 0.03 | 0.15 | 25 |
| byPeriod | 1655 | 1913 | 1250 |
| byFilingStatus | 3463 | 4128 | 50 (after `limit` on 1667 candidates) |

`byPeriod` and `byFilingStatus` blow past 250ms ‚Äî but that's
**fake-indexeddb's cost**, not the hub's. Comparing against the
`to-memory` baseline for the same 5K / 1250-row query:

| query | to-memory p95 | fake-indexeddb p95 | ratio |
|---|---:|---:|---:|
| byPeriod | 27 ms | 1913 ms | **~70√ó** |

The 1250 `adapter.get()` calls each go through a separate
`fake-indexeddb` transaction, and that's where the latency lives.
Chromium's native IDB typically clears a batch of `get()`s in the
same transaction in 0.1‚Äì1ms each ‚Äî making the 250ms gate
achievable in the runtime the issue actually targets.

## Where the acceptance artefact lives

The **1K smoke gate** passes in the bench harness above and is
asserted on every CI run that sets `NOYDB_BENCH=1`. It catches
catastrophic regressions in the query dispatch path.

The **50K √ó native-IDB** acceptance requires a Playwright-driven
Chromium run. That's deferred ‚Äî `fake-indexeddb` is not a sufficient
stand-in for the final pilot-2 sign-off, and porting the bench to
the showcase Playwright infra is tracked in the [#270 thread](https://github.com/vLannaAi/noy-db/issues/270).
When that lands, the smoke gate stays in the Node CI pipeline as
the first line of defence; the browser-side run becomes the
release-gate artefact.
