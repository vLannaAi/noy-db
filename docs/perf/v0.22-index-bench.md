# Lazy-mode index benchmark — v0.22 / v0.23 phase A

**Acceptance gate (issue [#270](https://github.com/vLannaAi/noy-db/issues/270)):**
p95 < 250ms on three pilot-2 queries at 50K records, cold start, on `to-browser-idb`.

## Current status

Node-side baseline on `to-memory` is **in place** (script:
`packages/hub/scripts/bench-lazy-indexes.mjs`).
The browser-side harness on `to-browser-idb` is a follow-up —
requires wiring through the showcase Playwright infrastructure so CI
can run it on every PR that touches query or index code.

## Running locally

```bash
pnpm --filter @noy-db/hub build
node packages/hub/scripts/bench-lazy-indexes.mjs           # full 50K
node packages/hub/scripts/bench-lazy-indexes.mjs 5000      # quick iteration
```

## Representative sample — 5,000 records, to-memory, Node 20.19.3

| query | p50 (ms) | p95 (ms) | n rows |
|---|---:|---:|---:|
| byClient | 0.02 | 0.06 | 25 |
| byPeriod | 23.74 | 27.39 | 1250 |
| byStatusOrdered | 32.06 | 35.85 | 50 |

Interpretation:

- `byClient` (high-selectivity equality) is effectively free — the
  mirror lookup + 25 AES-GCM decrypts finish in under 100µs.
- `byPeriod` is a 1250-row bucket → 1250 decrypts dominates. Scales
  linearly with bucket size. At 50K records × 4 periods ≈ 12,500
  matches per bucket, expect ~250ms on the node baseline.
- `byStatusOrdered` runs `orderedBy` on the status bucket then slices
  50 — wall time is dominated by the decrypt-and-filter pass over
  the 1,667-row candidate set (33% of records in 'submitted').

## Deferred: browser acceptance gate

The 250ms gate is specifically against `@noy-db/to-browser-idb` in a
real browser (Chromium via Playwright). Each `adapter.get()` there
pays a real IDB round-trip — the dominant cost at scale. The
showcase repo already has the Playwright wiring; this bench needs a
seed script that runs under Playwright + a telemetry capture pass.
Tracked in the comments on [#270](https://github.com/vLannaAi/noy-db/issues/270).
