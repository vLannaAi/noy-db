## 1. Scope & non-goals

### In scope

- Persistent, encrypted secondary indexes on lazy-mode collections, declared at collection construction via the existing `indexes: ['field', …]` option.
- Equality `.where(field, '==', value)` and `.in(field, [...])` dispatch through the index.
- `.orderBy(field, 'asc' | 'desc')` alone or composed with `.where(...)` on indexed fields.
- Composite queries require **both filter and order fields** to be indexed — throw a typed `IndexRequiredError` at query-build time otherwise.
- `.loadMore({ pageSize })` over sorted indexed results via an in-memory offset cursor.
- `collection.rebuildIndexes()` / `rebuildIndex(field)` / `reconcileIndex(field, { dryRun })` / `vault.processIndexReconcileQueue()` APIs.
- `noydb index rebuild/reconcile` CLI commands.
- 50K-record Playwright benchmark on `to-browser-idb` committed to `docs/perf/v0.22-index-bench.md`; p95 < 250 ms gate.

### Non-goals (deferred to v0.23.0)

- Range predicates (`<`, `<=`, `>`, `>=`, `between`).
- Composite / multi-field indexes (single index over `(clientId, period)`).
- Full-text search, prefix, substring match.
- Automatic reconciler on vault open — v0.22 is explicit-trigger only.

### Non-goals (permanent, by architectural invariant)

- Thai / locale-aware collation — userland per `docs/patterns/i18n-boundaries.md`.
- Adapter-level atomic batch writes — 6-method `NoydbStore` contract preserved.
- Adapter-visible per-value cardinality — explicitly prevented by the record-id scheme.

### Document touches

- `docs/reference/architecture.md` — extend "Caching and lazy hydration" section with indexes subsection; add threat model entry.
- `SPEC.md` — informative note on index envelope; no invariant changes.
- `CLAUDE.md` — update lazy-mode summary to note indexes are now allowed.
- `docs/spec/archive/issue-{260,265,266,267,268,269,270,271}.md` — written when each ticket closes.
- `docs/perf/v0.22-index-bench.md` — new, authored by PR 6.

## 2. Data layout + threat model

### 2.1 Envelope

Reuse the existing record envelope unchanged:

```json
{ "_noydb": 1, "_v": <int>, "_ts": "<iso>", "_iv": "<b64>", "_data": "<b64 ciphertext>" }
```

Encrypted body is:

```json
{ "value": <indexed field value, verbatim>, "writtenAt": "<iso>" }
```

`field` and `recordId` live in the record id, not the body — saves ~50 B per entry. The body is never read without its id, so the redundancy would be pure overhead.

### 2.2 Record id namespace

Reserved id prefix inside the same logical collection. Matches the existing `_ledger_deltas/<…>` / `_keyring` / `_meta/handle` convention — no changes to the 6-method store contract.

```
_idx/<field>/<recordId>
```

Example, disbursements collection with `clientId` indexed:

```
_idx/clientId/01HK3M…ABC   body → { value: "c-42", writtenAt: "2026-04-24T02:03Z" }
_idx/clientId/01HK3N…DEF   body → { value: "c-17", writtenAt: "2026-04-24T02:04Z" }
```

### 2.3 Storage footprint (Pilot-2 profile)

Per indexed field, per record: one envelope (~220 B fixed overhead + body size).

| Collection | Records | Indexed fields | Idx entries | ≈ disk | ≈ decrypted in-RAM |
|---|---:|---:|---:|---:|---:|
| disbursements | 10 K | 3 (clientId, period, filingStatus) | 30 K | ~8 MB | ~3 MB |
| invoice-scans (blob refs) | 125 K | 2 (clientId, period) | 250 K | ~65 MB | ~25 MB |

Decrypted index working set ≈ 25–30 MB — orders of magnitude below the main record budget.

### 2.4 Key hierarchy

Index envelopes are encrypted with the **collection DEK** (option b from [#265](https://github.com/vLannaAi/noy-db/issues/265)). No separate index DEK, no per-field DEK. Rationale:

- Anyone with the collection DEK already sees `value` in the main record — separating keys adds no confidentiality.
- Keyring stays uncluttered; rotation re-encrypts indexes during the standard collection re-encrypt pass.
- IV-budget at Pilot-2 scale (2³² encryption horizon) is safe with random 12-byte IVs.

### 2.5 Bulk-load model ("option X" from brainstorm)

- **Lazy load.** First `.query()` / `.where()` / `.orderBy()` against an indexed field triggers `store.list(vault, collection)`, which returns every id in the collection. The hub filters client-side to ids starting with `_idx/<field>/`, decrypts those envelopes, and populates an in-memory `PersistedCollectionIndex` (parallel of the eager-mode `CollectionIndexes` in `packages/hub/src/query/indexes.ts`). No store-interface change — the 6-method contract is preserved.
- **Session-persistent.** Once loaded, the index stays in memory for the life of the `Collection` instance. Not evicted by the lazy LRU — indexes have their own budget.
- **Invalidation.** Discarded on incoming sync changes, on `collection.close()`, on keyring rotation. Re-loaded on next access.

### 2.6 Threat model — what the adapter observes

| Observation | Visible? | Leakage assessment |
|---|---|---|
| Total count of `_idx/<field>/*` ids | yes | equals main record count — already visible via `list()`, no new leak |
| Per-record id of each index entry | yes | same recordId as the main record, no new leak |
| Which fields have indexes declared | yes (field names in the id path) | **only new metadata leak v0.22 introduces** — schema-level, declared by operator intent |
| Per-value cardinality (status distribution, per-client record counts) | **no** | central win of "option X" vs HMAC-prefix alternatives — value lives inside ciphertext |
| Ordering of values | no | value inside ciphertext; adapter-side sort is impossible |
| Write timing correlation | yes (clock-granularity) | no new leak beyond `_ts` on the main envelope |

### 2.7 Commitments the spec makes

- No plaintext field value leaves the hub.
- No value-derived key material is stored anywhere — no HMAC-of-value in ids, no bucket hashes.
- The index working set lives behind the same crypto boundary as the record cache (same collection DEK).
- Revoke-then-rotate semantics unchanged.

## 3. Write path + reconciler

### 3.1 `collection.put(id, record)` — indexed lazy flow

1. Compute main envelope (AES-256-GCM with collection DEK, fresh IV).
2. `store.put(vault, collection, id, mainEnvelope, expectedVersion?)` — main first.
3. If step 2 throws → rethrow; nothing else changed.
4. For each indexed field `f`:
   - `idxId = '_idx/' + f + '/' + id`
   - `idxBody = { value: readPath(record, f), writtenAt: mainEnvelope._ts }` (dotted-path field names inherit eager-mode `readPath` semantics)
   - `idxEnvelope = encrypt(idxBody, collectionDEK)` (fresh random 12-byte IV)
   - `try store.put(vault, collection, idxId, idxEnvelope)` — sequential, **no `expectedVersion`** (index writes are CAS-free; race-safety is via main-first ordering + read-time dangling filter) / catch → collect into `IndexWriteFailureError[]`; continue.
5. If `IndexWriteFailureError[].length > 0`:
   - `hub.indexReconcileQueue.add(collection, failedFields)` (in-memory, process-scoped — discarded if the process dies; `reconcileIndex` on next vault open is the cross-process recovery path).
   - Emit `index:write-partial` event with `{ recordId, failedFields, op: 'put' }`.
   - **Do not throw** — the record put succeeded; indexes are eventually-consistent on failure.
6. Upsert the in-memory mirror (`PersistedCollectionIndex`) for **every** field, including failed ones, so queries in this session remain correct.

### 3.2 `collection.delete(id)` — mirror-image flow

1. Read previous record (for mirror bookkeeping and to know which buckets to drop).
2. `store.delete(vault, collection, id)` — main first.
3. For each indexed field `f`: try `store.delete(vault, collection, '_idx/' + f + '/' + id)`; collect failures; continue.
4. Remove from in-memory mirror for every field.
5. On any failure: queue reconcile + emit `index:write-partial` with `op: 'delete'`.

### 3.3 Read-time dangling filter (safety net)

Every index-dispatched query runs a final pass:

```text
resolved = []
for each candidateRecordId yielded by the index intersection:
    main = await store.get(vault, collection, candidateRecordId)
    if (!main) {
        mirror.removeDangling(candidateRecordId)   // self-heal in-memory
        continue
    }
    resolved.push(decrypt(main))
```

- Makes false-positives (dangling index entries pointing to a gone record) invisible.
- Cost is amortized — `get()` is needed to return data anyway.
- **Concurrent redundancy note:** if a read and an async `reconcileIndex` run simultaneously, they may both remove the same dangling entry from the mirror. This is benign — `removeDangling` is idempotent. Spec acknowledges the redundancy for transparency.

### 3.4 `index:write-partial` event

```ts
collection.on('index:write-partial', ({ recordId, failedFields, op }) => { … })
```

- `op`: `'put' | 'delete'`.
- Consumers surface as non-fatal warnings or trigger out-of-band reconcile.
- Event routed through the existing Collection emitter — no new event channel.

### 3.5 Reconciler (#260-E)

**API:**

```ts
await vault.collection('disbursements').rebuildIndex('clientId')
await vault.collection('disbursements').rebuildIndexes()

const report = await vault.collection('disbursements').reconcileIndex('clientId', { dryRun: true })
// report: { missing: [{recordId, field, value}], stale: [{recordId, field}], applied: number }
await vault.collection('disbursements').reconcileIndex('clientId', { dryRun: false })

await vault.processIndexReconcileQueue()
```

**CLI:**

```bash
noydb index rebuild <vault> <collection> [<field>]
noydb index reconcile <vault> <collection> <field> [--dry-run] [--apply]
```

**Trigger model:** all explicit. No automatic run on vault open, no background timer. Operator decides when. Automatic on-open deferred to v0.23.0.

### 3.6 Mid-write failure classes

| Failure | Persistent state | Live session | Heals via |
|---|---|---|---|
| Main put fails | nothing changed | caller sees throw | — |
| Main put OK, ≥1 idx puts fail | record exists, some indexes missing | mirror correct this session; event emitted | `processIndexReconcileQueue()` or `reconcileIndex` |
| Main delete OK, ≥1 idx deletes fail | dangling entries point to gone record | mirror correct this session | read-time dangling filter (immediate) or `reconcileIndex` (eventual) |
| Process dies between main put and idx puts | record exists, no indexes | next session: record missing from queries | `reconcileIndex` on operator schedule |

### 3.7 Write throughput

- **Non-indexed collections:** `if (indexes.size === 0) return` early — zero overhead, within <5% target from #266.
- **Indexed collections:** linear in #indexed fields (N sequential adapter puts per record). On IDB with 3 indexes: ~4× main-write latency. Acceptable — benchmark harness measures the read path, which is the acceptance gate.

## 4. Read path

### 4.1 Query-planner lift

Replace the current `QueryRequiresEagerError` throw at `packages/hub/src/collection.ts:651–657`:

```text
if (lazy) {
  touchedFields = fields referenced in .where() / .orderBy()
  if (every touchedField ∈ this.indexes) → dispatch through PersistedCollectionIndex
  else → throw new IndexRequiredError({ collection, touchedFields, missingFields })
}
```

`IndexRequiredError extends NoydbError`, code `'INDEX_REQUIRED'`. Thrown at query-*build* time (on terminal `.toArray()` / `.first()` / `.loadMore()`), not at construction — matches the existing validation pattern.

### 4.2 Equality dispatch (#260-C)

1. Ensure `PersistedCollectionIndex` loaded (bulk-load once per session, then cached).
2. For each `.where(f, '==', v)` / `.where(f, 'in', [...])`: `ids_f = mirror.lookupEqual(f, v)` or `mirror.lookupIn(f, vs)`.
3. Intersect all `ids_f` sets → `candidateIds`.
4. For each id in `candidateIds`: `store.get()`; drop if dangling; decrypt otherwise.
5. MVP: all `.where()` clauses must reference indexed fields. Mixed indexed + unindexed → `IndexRequiredError`.
6. Increment `query._debug.indexHits++` for benchmark-harness consumption. Not public API.

### 4.3 orderBy dispatch (#260-D)

```text
.orderBy(f, dir) on indexed f:
  1. Ensure mirror loaded.
  2. sortedEntries = mirror.orderedBy(f, dir)   // Array<{recordId, value}> sorted by value
  3. If .where() filters present: intersect sortedEntries with candidateIds (set-membership test, preserves order)
  4. Return LoadMoreCursor wrapping resulting array + numeric offset.
```

`LoadMoreCursor.next({ pageSize })`: slice `offset..offset+pageSize`, parallel `get()`, dangling filter, advance offset.

### 4.4 Composite-query rule

Query with `.where(f1, …)` and `.orderBy(f2, …)` where `f1 ≠ f2`: both must be indexed. Violation → `IndexRequiredError` at build time with both fields in the payload.

Silent scan-fallback is explicitly rejected — it would hide the performance cliff v0.22 exists to prevent.

### 4.5 Ordering correctness

- **Numbers, Dates-as-Date:** Date coerced via `.toISOString()` before compare (consistent with eager-mode `stringifyKey`). Numeric compare uses native `<`/`>`.
- **Strings (incl. ISO dates):** default lexicographic. Chronological ISO-8601 compares correctly. No locale-aware collation — userland.
- **Booleans, null, undefined:** not supported as order keys; documented. Sorting a boolean index is undefined behavior.

### 4.6 Throughput budget

- **Cold first query:** `list(collection, '_idx/<field>/*')` + bulk decrypt → dominates cold latency. Measured as the worst-case path by the benchmark harness.
- **Warm queries:** mirror hit + O(matches) `get()` + decrypt. p95 < 250 ms on 50K records is the gate.

## 6. Benchmark + Pilot-2 sign-off

### 6.1 Harness location

`showcases/20-lazy-mode-index-bench/` — Playwright-driven, real Chromium, `to-browser-idb` backend. Matches existing `showcases/` conventions (numbered, self-contained, CI-runnable).

### 6.2 Dataset generator

Deterministic 50,000-record `disbursements` + 125,000-record `invoice-scans` generated in-run from a fixed PRNG seed. No persistent fixture file — ciphertext is KEK-specific.

### 6.3 Measured queries

| id | shape | indexed fields |
|---|---|---|
| byPeriod | `.where('period','==','2026-Q1').toArray()` | `period` |
| byClient | `.where('clientId','==','c-42').toArray()` | `clientId` |
| byFilingStatus | `.where('status','==','submitted').orderBy('filedAt','desc').loadMore({pageSize:50})` | `status`, `filedAt` |

100 runs each, p50 + p95, cold (fresh session, first query) and warm (subsequent).

### 6.4 Baseline comparison

Same queries against a **non-indexed collection** at the same dataset size — uses `collection.scan()` + userland filter (the only alternative without indexes). Reports speedup ratio.

### 6.5 Report (`docs/perf/v0.22-index-bench.md`)

Sections: environment (CPU, browser version), dataset description, per-query table (p50/p95 cold/warm, index vs scan), disk footprint, final sign-off stamp.

### 6.6 CI integration

`.github/workflows/ci.yml` gains a `bench-indexes` job that runs on any PR touching `packages/hub/src/query/` or `packages/hub/src/collection.ts`. Fails if p95 regresses >10% over the committed baseline. Non-blocking on doc-only PRs.

### 6.7 Acceptance criteria (triggers PR 7)

- All three queries p95 < 250 ms at 50K records, cold, on `to-browser-idb`.
- Warm p95 < 50 ms (mirror-hit sanity).
- Scan baseline ≥10× slower on at least 2 of 3 queries (proves index value).
- Report committed, dated, signed off by Pilot-2 reviewer in PR body.
