## Summary

Second v0.4 feature: **hash-chained audit log**. Every \`Collection.put\` and \`Collection.delete\` now appends an encrypted entry to the compartment's \`_ledger/\` collection. Entries are linked by \`prevHash = sha256(canonicalJson(previousEntry))\` so any tampering breaks the chain at that point and is detected by \`ledger.verify()\`.

## API

\`\`\`ts
const company = await db.openCompartment('demo-co')
const invoices = company.collection('invoices')

await invoices.put('inv-1', { /* ... */ })
await invoices.delete('inv-1')

const ledger = company.ledger()
const head = await ledger.head()
// → { entry: {...}, hash: 'a1b2c3...', length: 2 }

const verifyResult = await ledger.verify()
// → { ok: true, head: 'a1b2c3...', length: 2 }
// or { ok: false, divergedAt: 5, expected: '...', actual: '...' }
\`\`\`

## Design highlights

1. **Zero-knowledge preserved.** \`payloadHash\` is the sha256 of the encrypted envelope's \`_data\` field, NOT the plaintext. The full ledger entry is itself encrypted with a per-compartment ledger DEK, so adapters never see plaintext metadata even though they hold the ledger entries.

2. **Cached head for O(1) appends.** Each LedgerStore caches its head in memory so \`append()\` doesn't re-scan the adapter on every call. **The first version of this PR was O(N²) and the test suite caught it** — a 1K-record stress test went from <100ms to a 5-second timeout. The fix turns it into one canonical-JSON encode + one sha256 per append.

3. **System-collection DEK propagation in \`grant()\`.** Side-effect fix: the keyring grant flow now propagates ALL system-prefixed collection DEKs (\`_ledger\`, \`_history\`, \`_sync\`) to every grant target, not just owner/admin/viewer roles. Required so an operator with write access on a single collection can still append to the compartment's shared ledger. Trade-off (operator can decrypt ledger entries about other collections, which is metadata-not-plaintext) documented inline.

4. **Hidden from \`loadAll\`.** \`_ledger\` follows the existing \`_keyring\` / \`_history\` / \`_sync\` convention — backups and exports filter underscore-prefixed collections out automatically.

Coverage:
- \`canonicalJson\` sort-stability across object key insertion order
- Primitive encoding parity with \`JSON.stringify\`
- Rejection of \`undefined\`, \`NaN\`, \`Infinity\`, \`BigInt\`, functions
- \`sha256Hex\` determinism + format
- \`hashEntry\` = \`sha256Hex(canonicalJson(entry))\`
- \`Collection.put\` appends a put entry with the right shape
- \`Collection.delete\` appends a delete entry with the deleted version
- Sequential indices + chained \`prevHash\`
- Multi-collection compartments share one chain
- \`verify()\` returns \`ok: true\` on a clean chain
- \`verify()\` rejects mid-chain ciphertext tampering (AES-GCM auth tag fails on decrypt)
- \`verify()\` rejects forged appends from outside the LedgerStore
- 100-record stress test (perf gate, would have caught the O(N²) regression)
- Cross-instance \"process restart\" rehydrate from adapter
- \`head().hash\` is stable for external anchoring use

## What's NOT in this PR (deferred)

- **Delta history via JSON Patch** — paired follow-up (#44 — next PR)
- **Grant/revoke/rotate as ledger operations** — only put/delete are recorded today; access ops need a separate hook in the keyring path
- **Merkle proofs** for individual entries — nice-to-have, post-v0.4
- **\`verifyIntegrity()\`** cross-check between ledger and data collections — catches the case where someone writes to a data collection bypassing the LedgerStore entirely
- **Multi-writer hardening** — single-writer is the v0.4 assumption; documented in the LedgerStore docstring
