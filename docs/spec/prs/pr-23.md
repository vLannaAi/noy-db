## Summary

Adds the **second adoption path** for v0.3 Pinia integration: a Pinia plugin (\`createNoydbPiniaPlugin\`) that lets existing \`defineStore()\` stores opt into NOYDB persistence by adding a single \`noydb:\` option, with **no component code changes**.

Together with #10 (\`defineNoydbStore\` greenfield path, already merged as feb1ef5), this completes the v0.3 Pinia integration surface. The two together cover the entire adoption story:

| Project type | Path | Adoption cost |
|--|--|--|
| Greenfield | \`defineNoydbStore\` | One line per store |
| Existing Pinia app | \`createNoydbPiniaPlugin\` + \`noydb:\` option | Zero component changes |

\`\`\`ts
import { createPinia } from 'pinia';
import { createNoydbPiniaPlugin } from '@noy-db/pinia';
import { jsonFile } from '@noy-db/file';

const pinia = createPinia();
pinia.use(createNoydbPiniaPlugin({
  adapter: jsonFile({ dir: './data' }),
  user: 'owner-01',
  secret: () => promptPassphrase(),
}));

// existing store — add ONE option, no component changes:
export const useClients = defineStore('clients', {
  state: () => ({ list: [] as Client[] }),
  noydb: { compartment: 'C101', collection: 'clients', persist: 'list' },
});

## Notes for reviewers

### The Pinia plugin testing gotcha

The most subtle bug in this PR was in the test setup, not the plugin code: **Pinia plugins only fire when the Pinia instance is mounted on a Vue app via \`app.use(pinia)\`**. Just calling \`setActivePinia()\` is NOT enough — the app context is what triggers plugin instantiation. The test helper \`makePinia()\` documents this and creates a stub Vue app for every plugin test. Caught this with a 5-line debug test once the symptom (\`\$noydbAugmented\` undefined for every store) became clear.

### Single-document persistence model

The augmentation plugin persists each augmented store as **one document** at id \`__state__\` containing the picked state subset. This is different from \`defineNoydbStore\` which maps individual records into the collection. The reason: when augmenting an existing store, you don't know if \`state.list\` is meant as "an array of records" or "a single state field that happens to be an array". The single-document model is unambiguous and matches what \`pinia-plugin-persistedstate\` does.

### Lazy Noydb instantiation

The Noydb instance is constructed on first use, not on plugin install. Apps with no \`noydb:\`-augmented stores pay **zero crypto cost** — secret() is never called, no PBKDF2 derivation, no keyring read. Verified by test #13 (\`secret never called when no augmented stores\`).

### \$noydbFlush()

Added because fire-and-forget puts make tests racy. Now both tests AND app code can deterministically wait for persistence to complete. Useful before unmounting components that just mutated the store. Documented in the README.
