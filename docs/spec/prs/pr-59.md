## Summary

New workflow \`.github/workflows/e2e-scaffolder.yml\` that reproduces the consumer's perspective on every PR touching the scaffolder or its runtime deps. This is the CI job we've been promising ourselves since the v0.3.1 → v0.3.2 patch, and it would have caught all three known historical release-time bugs.

## What it does

For each matrix cell (OS × Node × PM), the job:

1. Builds \`@noy-db/{core,memory,browser,pinia,create}\` via turbo
2. Packs each one with \`pnpm pack\`
3. Creates a fresh consumer directory under \`$RUNNER_TEMP\` — **outside the workspace**, so no accidental hoisting
4. Installs the tarballs with the matrix's package manager
5. Invokes the \`create\` bin to scaffold a Nuxt 4 starter
6. Runs an end-to-end smoke test that imports and exercises every major v0.4 export
7. Runs \`noy-db verify\` as a final check

## Matrix

- **OS**: \`ubuntu-latest\`, \`macos-latest\`
- **Node**: 20, 22
- **Package manager**: npm, pnpm

**8 jobs per run**, \`fail-fast: false\`.

Windows is deferred (PM tooling quirks produce flakes without adding signal). Bun and Yarn are out of scope for now — npm and pnpm exercise the two most different resolution models (npm's strict peer deps vs pnpm's isolated virtual store) and cover the vast majority of real-world install paths.

## Path filter

Only runs when something that could affect the consumer install path changes:

\`\`\`yaml
paths:
  - 'packages/create-noy-db/**'

  - 'packages/browser/**'
  - '.github/workflows/e2e-scaffolder.yml'
\`\`\`

Plus all pushes to \`main\` for belt-and-braces.

This job would have caught all three known historical release-time bugs:

| Bug | How this job catches it |
|---|---|
| **v0.3.1 → v0.3.2**: \`@noy-db/core\` and \`@noy-db/memory\` declared as \`devDependencies\` of \`@noy-db/create\` → \`noy-db verify\` threw \`ERR_MODULE_NOT_FOUND\` | Smoke test imports both packages directly; install step would fail |
| **v0.4.0 → v0.4.1**: \`workspace:*\` in \`peerDependencies\` published as pinned exact-version strings → \`npm install @noy-db/core@0.4.0 @noy-db/memory@0.3.0\` produced \`ERESOLVE\` | Install step installs all tarballs together; any ERESOLVE surfaces as non-zero exit |
| **Hypothetical**: files missing from the \`files\` list → workspace sees them, published tarball doesn't | Smoke test imports every major v0.4 symbol; missing file → import throws |

## Design notes called out in the workflow file

The workflow comments explain the load-bearing decisions inline:

- **Why the consumer dir lives outside the workspace** — pnpm/npm walk up looking for a workspace root; if we put it inside, they find ours and silently fall back to the source. \`$RUNNER_TEMP\` is outside.
- **Why tarball paths are resolved once via a shared step output** — the paths contain the version string, which can change between runs, so we glob-resolve once and pass the paths through \`$GITHUB_OUTPUT\` so every downstream step sees the same files.
- **Why we install \`pinia\` + \`vue\` as explicit top-level deps** — they're peer deps of \`@noy-db/pinia\`, and pnpm's isolation model requires they live in the consumer's top-level deps. npm would hoist them automatically but we want parity across the matrix.
