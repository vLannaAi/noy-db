## Summary

The v0.3.1 publish attempt earlier tonight failed with **403 Forbidden** on \`PUT https://registry.npmjs.org/create-noy-db\`. Root cause: this repo's \`NPM_TOKEN\` is a granular automation token scoped to the \`@noy-db\` org. It has create-package rights inside that scope (which is why every \`@noy-db/*\` package publishes cleanly) but **cannot create new unscoped packages**.

## Fix

Publish the package inside the scope as \`@noy-db/create\` instead of \`create-noy-db\`. Users invoke via npm's scoped-initializer convention:

\`\`\`bash
npm  create @noy-db my-app
pnpm create @noy-db my-app
yarn create @noy-db my-app
bun  create @noy-db my-app
\`\`\`

This works because \`npm init @scope\` is sugar for \`npm exec @scope/create\`, which in turn runs the bin matching the stripped scope name (\`create\`). Same convention as:

- \`@vitejs/create-app\` → \`npm create @vitejs/app\`
- \`@nrwl/create-nx-workspace\` → \`npm create @nrwl/nx-workspace\`

## Trade-offs considered

**Alternative A** — Generate a new wider-scoped NPM_TOKEN with "write any package" permission. Cleaner invocation (\`npm create noy-db@latest\`) but requires the user to:
1. Log into npmjs.com
2. Create a new granular token
3. Update the \`NPM_TOKEN\` secret in the GitHub repo

**This fix (Alternative B)** — Rename to \`@noy-db/create\`, no token changes, publishes fire from existing CI setup. Invocation is \`npm create @noy-db\` (slightly different but standard and widely used).

I went with (B) because it's a code-only change I can deliver in-band, and the \`npm create @noy-db\` convention is well-established. If you later generate a wider token, we can add an unscoped alias package that's just a shim.
