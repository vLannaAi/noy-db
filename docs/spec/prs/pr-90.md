## Existence-leak guarantee

\`listAccessibleCompartments()\` walks the compartment universe via the adapter, then for each compartment attempts to load the calling user's keyring with the in-memory passphrase. Compartments where the user has no keyring file (\`NoAccessError\`) or where the passphrase doesn't unwrap (\`InvalidKeyError\`) are silently dropped — **the existence of those compartments is never confirmed in the return value**.

The adapter sees the enumeration call (it has to — it owns the storage), but downstream consumers of \`listAccessibleCompartments()\` only see the filtered list. That's the boundary the existence-leak guarantee draws.

## queryAcross design notes

- **Per-compartment errors do not abort the fan-out.** If one compartment's callback throws, that compartment's slot carries the error and the remaining compartments still run. The caller decides how to handle the partition between success and failure. This is the right shape for cross-tenant reports where one tenant's outage shouldn't hide the other tenants' data.
- **Concurrency is opt-in.** Default is \`1\` (sequential) — conservative because per-compartment callbacks typically do their own I/O and an unbounded fan-out can exhaust adapter connections. Bump to 4-8 for cloud adapters where parallelism is the whole point.
- **The scheduler is inline.** ~25 lines of p-limit-style logic, no external dep. Maintains a sliding window of \`concurrency\` in-flight tasks via \`Promise.race\` against a live set, and writes results into a pre-allocated index-keyed array so caller-supplied order is preserved regardless of which task finishes first.

## Composes with exportStream() — cross-compartment plaintext export

```ts
const accessible = await db.listAccessibleCompartments({ minRole: 'admin' })
const exports = await db.queryAcross(
  accessible.map((c) => c.id),
  async (comp) => {
    const out: unknown[] = []
    for await (const chunk of comp.exportStream()) out.push(chunk)
    return out
  },
)
```

This is one of the load-bearing reasons \`exportStream()\` lives in core (#72). The moment you have both primitives, the cross-tenant export story falls out without any new code.

## Known v0.4 edge case (documented, not fixed in this PR)

A compartment whose keyring file happens to have an empty wrapped-DEKs map (because the owner granted access *before* any collection was created) will pass the \`loadKeyring\` probe with **any** passphrase — there are no DEKs to unwrap, so the integrity-checked unwrap that normally rejects wrong passphrases never runs. The result: an unrelated principal who knows the user-id and the compartment name can show up in \`listAccessibleCompartments()\` as having access to that empty compartment.

This is a metadata leak (compartment name + user-id), **not a content leak** — the principal cannot read any actual data because their DEK set is empty. Hardening via a passphrase canary in the keyring file format is tracked as a v0.6+ follow-up. The limitation is documented in the \`listAccessibleCompartments()\` JSDoc.

## Cost analysis

\`listAccessibleCompartments\` is O(compartments × keyring-load) — one \`loadKeyring\` attempt per compartment in the universe. Each attempt does one adapter \`get\` + one PBKDF2 derivation (600K iterations) + N AES-KW unwraps. For dozens of compartments this is fine; for thousands the consumer should cache the result and refresh on grant/revoke events.

Small bonus: every compartment whose keyring is successfully unwrapped during the probe is **opportunistically primed in the keyring cache**, so a subsequent \`openCompartment(id)\` against any of those compartments doesn't have to re-derive the KEK. The cost is one Map.set per compartment we already paid to unwrap.

## Insight for reviewers

- The adapter capability check uses \`typeof adapter.listCompartments !== 'function'\` rather than a registry or capability flag. This matches the existing \`listPage\` pattern in the codebase and means downstream adapter authors don't have to register anywhere — implementing the optional method is the entire opt-in. The runtime cost of the check is one property access per call, which is unmeasurable.
- The opportunistic keyring-cache priming during enumeration is a small but real win. Without it, a typical \"enumerate then queryAcross\" pattern would derive each KEK twice — once during the probe, once during \`openCompartment\` inside the callback. With it, the probe pays the cost and the open call hits the cache. For 600K-iteration PBKDF2, this matters: 50ms × N compartments saved per cross-compartment fan-out.
- The empty-compartment edge case is the kind of v0.4 hardening item that only surfaces when you write a test that's specifically designed to exercise the wrong-passphrase rejection and watch it fail. Worth filing the v0.6+ follow-up for the passphrase canary while the context is fresh.
