## Summary

Adds the chainable, immutable query DSL to `@noy-db/core`. Filtering runs client-side after decryption — preserves zero-knowledge.

```ts
const overdue = invoices.query()
  .where('status', '==', 'open')
  .where('dueDate', '<', new Date())
  .orderBy('dueDate', 'asc')
  .limit(50)
  .toArray();
```

## Notes for reviewers

### Variance design

The internal \`Clause\`, \`QueryPlan\`, \`FilterClause\`, \`GroupClause\` types are intentionally **non-parametric** — they operate on \`unknown\` records at runtime. The \`T\` tag lives only at the public \`Query<T>\` API surface. This was necessary because \`FilterClause\` has a function input \`(record: T) => boolean\`, which would otherwise make \`Collection<T>\` invariant in T and break the \`Compartment.collection<T>()\` cache (which stores \`Collection<unknown>\`).

User-supplied predicates are cast at the boundary in \`filter(fn)\`. This is safe because the runtime never inspects T anyway.

### Tree-shaking

Collection still imports \`Query\` directly (the cost of \`query()\` being a method). If we wanted strict tree-shaking, we'd need to make \`query\` a free function instead. For v0.3 the +5 KB is acceptable. Revisit at v1.0 if bundle size matters.

### Compromises

- \`.live()\` from the issue spec is implemented as the framework-agnostic \`.subscribe(cb)\` here. The \`Ref<T[]>\`-returning \`.live()\` is a Vue-specific helper and lives in #10.
- Index-aware execution is gated on #13 — until then, every query is a linear scan over the in-memory cache. Fine for the v0.2 target scale (≤50K records).
