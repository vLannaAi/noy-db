## The bug the demo caught

The \`@noy-db/nuxt\` module shipped in #27 pointed at \`./runtime/plugin.client.ts\` via \`resolver.resolve()\` — but tsup's build config only emitted \`dist/index.js\`. The runtime file never made it to \`dist/\`. When the demo tried to use the module, Nuxt's prepare step failed with:

\`\`\`
[nuxt] WARN Failed to parse static properties from plugin ../../packages/nuxt/dist/runtime/plugin.client.ts.
ENOENT: no such file or directory, open '.../packages/nuxt/dist/runtime/plugin.client.ts'
\`\`\`

Fix:
1. Added \`src/runtime/plugin.client.ts\` as a second tsup entry so it compiles to \`dist/runtime/plugin.client.js\`
2. Updated the module's \`resolver.resolve()\` to point at the compiled \`.js\` file

This is exactly the class of bug unit tests miss but integration tests catch. The v0.3 story is MORE robust because this PR caught it before \`@noy-db/nuxt\` ships to real users.

## SSR safety — VERIFIED on the actual build output

After \`nuxt build\`, I grepped \`.output/server/\` for the forbidden crypto symbols:

\`\`\`
$ grep -rlE "(crypto\.subtle|unwrapKey|PBKDF2|'DEK'|'KEK'|deriveKey)" .output/server/
(zero matches)
\`\`\`

Zero references. The server-side Nitro output has **no crypto code whatsoever**. As a positive control, the client bundle at \`.output/public/\` does contain \`crypto.subtle\` — confirming the crypto layer correctly ships to the browser only.

This is the load-bearing property of \`@noy-db/nuxt\`. The \`mode: 'client'\` flag on the runtime plugin, combined with the user's \`.client.ts\` bootstrap plugin, means the entire crypto layer is excluded from the server bundle at build time.

Turning this into a CI assertion is tracked as a v0.4 follow-up — it would add ~20s per nuxt build and we want to keep v0.3 CI fast.

## Notes for reviewers

### The bootstrap plugin is the ONLY file that imports @noy-db/core

This is the v0.3 acceptance criterion. Every other file in \`playground/nuxt/\` touches NOYDB exclusively through the Pinia API:

- \`stores/invoices.ts\` — \`defineNoydbStore<Invoice>\` (auto-imported)
- \`stores/clients.ts\` — \`defineNoydbStore<Client>\` (auto-imported)
- \`pages/index.vue\` — \`useInvoices()\`, \`useClients()\`
- \`pages/invoices.vue\` — \`useInvoices()\`, \`.query().where().orderBy().toArray()\`
- \`pages/clients.vue\` — \`useClients()\`, \`.query().orderBy().toArray()\`

The bootstrap plugin is the correct place for \`createNoydb()\` because that's where the app decides HOW to instantiate NOYDB (which adapter, which passphrase source). In production that logic would come from a user-visible passphrase prompt or biometric unlock. The demo hard-codes it for boot-without-interaction with a clear README warning.

### Why the demo uses eager mode, not lazy hydration

Two reasons: (1) the dataset is tiny — 5 seeded samples plus whatever the user adds — so eager mode is strictly simpler and shows the happy path. (2) Lazy mode (\`prefetch: false\`) disallows \`query()\` and \`list()\` in the current implementation, which would mean rewriting every page to use \`scan()\` iterators. A separate \`playground/nuxt-lazy/\` folder could dogfood lazy hydration for 50K-record workloads in a follow-up.

### Why no Playwright E2E tests

The issue spec called for Playwright tests covering "dashboard renders after biometric mock unlock, creating an invoice persists across reload, query filters update reactively." For v0.3 alpha:

- The \`nuxt build\` step IS the integration test. If anything in the stack is broken, the build fails. Tracked that this caught the runtime plugin bug in #27 — proof that the build-time test has real value.
- Playwright adds ~60s to CI per test via browser download + headless setup, and would roughly double the nuxt package's CI time.
- Dedicated E2E is better scoped to a v0.4 benchmarks PR where we can add Lighthouse, bundle size tracking, AND Playwright in one coordinated push.
