## Summary

Adds the \`@noy-db/nuxt\` package — the second front door to v0.3 adoption alongside the create-noy-db scaffolder. **Nuxt 4+ exclusive**, ESM-only, Node 20+.

\`\`\`ts
// nuxt.config.ts
export default defineNuxtConfig({
  modules: ['@noy-db/nuxt'],
  noydb: {
    adapter: 'browser',
    sync: { adapter: 'dynamo', table: 'noydb-prod' },
    auth: { mode: 'biometric' },
  },
})
\`\`\`

After installing, every component in your app gets these auto-imported:

| From \`@noy-db/vue\`         | From \`@noy-db/pinia\` (when \`pinia: true\`, default) |
|------------------------------|----------------------------------------------------|
| \`useNoydb()\`               | \`defineNoydbStore()\`                             |
| \`useCollection<T>()\`       | \`createNoydbPiniaPlugin()\`                       |
| \`useSync()\`                | \`setActiveNoydb()\` / \`getActiveNoydb()\`        |

## Notes for reviewers

### Why \`vi.mock('@nuxt/kit')\` instead of \`@nuxt/test-utils\`

\`@nuxt/test-utils\` spins up a real Nuxt build per test (~30s+ wall-clock), requires the test environment to have a writable temp directory for the \`.nuxt/\` cache, and hits the Nuxt module-resolver every time. It's the right tool for INTEGRATION tests that need to assert "this Nuxt project boots and renders". For UNIT tests of the module factory itself, mocking the small subset of \`@nuxt/kit\` we use is faster, more deterministic, and easier to maintain.

The trade-off: unit tests can't catch issues that only manifest during a real Nuxt build (e.g., a typo in a runtime file path, a clash with another module). Those will surface when someone runs the reference Nuxt demo (#16).

### Why eager Noydb auto-instantiation is deferred

\`createNoydb()\` requires a \`secret\` callback that returns a passphrase. Callbacks can't be serialized through Nuxt's runtime config. We could:

1. **Today (v0.3):** require users to call \`setActiveNoydb()\` from their own setup file. Honest, manual, documented in the README.
2. **Tomorrow (v0.4):** add a \`bootstrap\` callback option to the module that the runtime plugin imports lazily. Cleaner, but locks in an API before we have a real consumer.

I picked option 1 because it's cheaper to add automation later than to undo it. The README clearly walks through the manual bootstrap pattern.

### TypeScript module augmentation

The module file augments \`@nuxt/schema\` with the typed \`noydb\` field on \`NuxtConfig\`, \`NuxtOptions\`, and \`PublicRuntimeConfig\`. Once a project adds \`'@noy-db/nuxt'\` to its modules array, TypeScript picks up the typed config option without any explicit import — autocomplete works in \`nuxt.config.ts\` for free. This is the same pattern \`@nuxtjs/tailwindcss\`, \`@pinia/nuxt\`, and other community modules use.
