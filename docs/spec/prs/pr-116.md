## Summary

- Multi-FK chaining: chain multiple `.join()` calls on the same query, each resolving an independent FK declared via `ref()`. Closes #75.
- Each leg picks its own planner strategy independently — a query can mix nested-loop and explicit hash join in the same chain.
- Each leg enforces its own ref-mode behavior independently (strict on one field, warn on another, no bleed).
- Tightens the per-leg left-side ceiling check so a later leg with stricter `maxRows` correctly fires on the same left set.

## Stacked on #73

This PR is stacked on `feat/v0.6/eager-join-73` (PR #115). Merge order: #115 → main, then this PR rebases onto main and merges. The runtime support for multi-leg chains already existed in the #73 executor (it loops over `plan.joins`); this PR is mostly the per-leg ceiling tightening + tests + changeset.

## API

```ts
const rows = invoices.query()
  .where('status', '==', 'open')
  .join<'client', Client>('clientId', { as: 'client' })
  .join<'category', Category>('categoryId', { as: 'category' })
  .toArray()
// → [{ id, amount, client: { ... }, category: { ... } | null }, ...]
```

## What changed in this PR

- **`packages/core/src/query/join.ts`** — moved the left-side ceiling check from a one-shot top-of-`applyJoins` block (using `joins[0].maxRows`) into `applyOneJoin`, so each leg enforces its own `maxRows` against the current left-row count. The previous shape silently ignored tighter ceilings on later legs.
- **`packages/core/__tests__/query-join-multi.test.ts`** — 8 new tests covering 2-join chains, 3-join shapes, mixed strategies, mixed ref modes, the per-leg left ceiling check, the equi-join 1:1 left-cardinality property, and `toPlan()` partition seam introspection.
- **`.changeset/multi-fk-chaining-75.md`** — minor bump.
