## Summary

`Collection.scan()` now returns a new `ScanBuilder<T>` that implements `AsyncIterable<T>` (backward-compatible `for await`) and exposes chainable `.where()` / `.filter()` clauses plus an async `.aggregate(spec)` terminal that reduces the scan stream through the same reducer protocol as `Query.aggregate()` (#97) — with **O(reducers) memory**, not O(records).

```ts
// Backward-compatible iteration — unchanged
for await (const record of invoices.scan({ pageSize: 500 })) {
  await processOne(record)
}

// v0.6 #99 — streaming aggregation with filter
const { total, n } = await invoices.scan({ pageSize: 1000 })
  .where('year', '==', 2025)
  .aggregate({ total: sum('amount'), n: count() })
```

## Design choices

**Memory bound is a code-level invariant.** The aggregate terminal initializes one mutable state per reducer, iterates through the stream one record at a time via `for await`, applies every reducer's `step` per record, and never collects the stream into an array. No runtime assertion needed — the bound is visible by reading the function body.

**Reducer reuse is the whole point.** Every factory from #97 (` + "`count`, `sum`, `avg`, `min`, `max`" + `) plugs into `scan().aggregate()` unchanged, and the `{ seed }` parameter plumbing from #87 constraint #2 is honored transparently. No duplicated API — the `Reducer<R, S>` protocol was deliberately designed so both `Query` and `Scan` aggregations could share it.

**Backward compatibility:** return type of `Collection.scan()` changed from `AsyncIterableIterator<T>` to `ScanBuilder<T>`. Every existing `for await (const rec of collection.scan())` call continues to work because `ScanBuilder` implements `[Symbol.asyncIterator]`. A pre-change grep confirmed zero call sites use `.next()` directly on the iterator. **All 36 existing pagination + lazy-hydration tests pass without modification.**

**Immutable builder** — each `.where()` / `.filter()` returns a fresh builder sharing the same page provider and page size. Base scans can be safely reused across multiple parallel aggregations, though each still pays a full scan. Multi-way single-pass aggregation is out of scope for v0.6 — consumers should build a compound spec.

**Clause evaluation on a scan does not use the secondary-index fast path** — the stream sources records from the adapter paginator, not from the in-memory cache where indexes live. Clauses run O(1) per record per clause in insertion order, short-circuiting on first false. Index-accelerated scans would need a new adapter API.

## Out of scope (tracked separately)

- `scan().aggregate().live()` — reconciling unbounded streaming with change-stream subscriptions is a design problem, not a code one. Consumers should narrow with `.where()` enough to fit in the 50k `query()` limit and use `query().aggregate().live()` instead.
- `scan().groupBy().aggregate()` — high-cardinality grouping on huge collections re-introduces the O(groups) memory problem that streaming aggregate was designed to avoid.
- Parallel scan across pages — no race-safe cursor contract in the adapter API yet.
- `scan().join(…)` — tracked under #76 streaming join.
- Index-accelerated scans.
