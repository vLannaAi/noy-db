## Summary

**Final v0.4 feature**: verifiable backups. \`dump()\` embeds the current ledger head + internal collections; \`load()\` re-runs verification after restoring and rejects any backup whose chain or data has been tampered with between dump and restore.

## API

\`\`\`ts
const company = await db.openCompartment('demo-co')
const invoices = company.collection('invoices')
await invoices.put('inv-1', { /* ... */ })

// Dump now embeds ledgerHead + internal collections (chain replay data)
const backupJson = await company.dump()

// Load runs verifyBackupIntegrity() after restoring
await company.load(backupJson)
//   → throws BackupLedgerError if the chain is broken or the head doesn't match
//   → throws BackupCorruptedError if any data envelope's payloadHash diverged

// Can be called any time on a live compartment, not just on load
const result = await company.verifyBackupIntegrity()
// → { ok: true, head, length }
// → { ok: false, kind: 'chain', divergedAt, message }
// → { ok: false, kind: 'data', collection, id, message }
\`\`\`

## What this PR detects

| Attack | Detection |
|---|---|
| Modify a ledger entry's encrypted bytes | AES-GCM auth tag fails on decrypt → throws |
| Reorder ledger entries | \`prevHash\` chain breaks → BackupLedgerError |
| Modify a data envelope's encrypted bytes | sha256 mismatch with ledger payloadHash → BackupCorruptedError |
| Modify the embedded \`ledgerHead.hash\` to match a tampered chain | Reconstructed head ≠ embedded → BackupLedgerError |
| Out-of-band write to a data collection (bypassing Collection.put) | Same data envelope cross-check catches it on the next \`verifyBackupIntegrity()\` call |

## Side fix: keyring reload on \`Compartment.load()\`

**Encrypted backups previously couldn't round-trip in a single process.** The old \`load()\` restored a different keyring file but the in-memory \`Compartment.keyring\` still held the pre-load session's DEKs, so every post-load decrypt failed with \`TamperedError\`. The fix:

- New \`reloadKeyring\` callback wired through Noydb → Compartment that re-derives the unlocked keyring from the user's passphrase against the freshly-loaded keyring file
- \`Compartment.load()\` calls it after restoring keyrings, then rebuilds the cached \`getDEK\` resolver so the next encrypted operation sees the loaded wrapped DEKs
- Noydb's \`keyringCache\` is also invalidated so future \`openCompartment\` calls see the refreshed keyring

This makes encrypted dump→load round-trips work for the first time. Plaintext compartments are unaffected (no callback is provided).

## What's NOT in this PR (deferred)

- **HMAC signing** — skipped for v0.4 because the chain itself is already the integrity guarantee. Adding a signature would protect against backup substitution (replacing the file with a different valid backup) which is out of scope for v0.4 and needs a key derivation story.
- **Per-collection granular integrity reports** — only the first failure is surfaced.
- **A \`verifyBackupIntegrity()\` analogue for old-format backups** — pre-v0.4 backups have no chain to verify against.
