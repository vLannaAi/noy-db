## Summary

**v0.4.0 shipped broken** — every \`@noy-db/*\` package pinned its \`peerDependencies\` to the exact workspace version via \`workspace:*\`, so consumers couldn't mix versions. Dogfooding the public \`v0.4.0\` install caught it immediately:

\`\`\`
$ npm install @noy-db/core@0.4.0 @noy-db/memory
npm error ERESOLVE
npm error Found: @noy-db/core@0.4.0
npm error Could not resolve dependency:
npm error peer @noy-db/core@"0.3.0" from @noy-db/memory@0.3.0
\`\`\`

Root cause: \`workspace:*\` in \`peerDependencies\` expands on publish to the **exact** version (\`"0.3.0"\`), not a caret range (\`"^0.3.0"\`). Every adapter published at v0.3 pinned its peer on core 0.3.0. When core jumped to 0.4.0, the adapters became uninstallable alongside it.

## Fix

1. **Every \`workspace:*\` in \`peerDependencies\` → \`workspace:^\`.** Publishes as caret range, letting consumers install any semver-compatible version.
2. **Every \`@noy-db/*\` package bumped to 0.4.1**, unifying the line. Previously mixed across 0.2.0 / 0.3.0 / 0.3.2 / 0.4.0.
3. **\`create-noy-db\` template dep specs bumped to ^0.4.1.**
4. **No functional code changes** — this is a manifest-only release.

## Side lesson codified

\`workspace:*\` is correct for \`dependencies\` (siblings always ship together from the same workspace commit) but **wrong** for \`peerDependencies\` (consumers can and will mix versions). The release procedure in \`CONTRIBUTING.md\` should be updated to warn about this — tracked as a separate follow-up.

This is the third release-time bug this epic has caught via the "pack + install in fresh dir" discipline (after v0.3.1's runtime-deps and scope-token issues). Worth keeping.
