## Summary

- Ships `Query.join(field, { as })` — resolves a `ref()`-declared foreign key into an attached right-side record under an alias. Closes #73.
- Two planner strategies auto-selected: nested-loop via `lookupById` (O(1) per left row, the common path for Collection targets) and hash join as a fallback for custom `QuerySource` implementations. Manual `{ strategy: 'hash' | 'nested' }` for tests.
- Hard 50k-per-side row ceiling via `JoinTooLargeError`, 80% warn, ref-mode behavior (`strict` → `DanglingReferenceError`, `warn` → null + one-shot warning, `cascade` → silent null). Same-compartment by construction.
- Honors #87 constraint #1 — every `JoinLeg` carries `partitionScope: 'all'` plumbed through the internal representation, ready for v0.10 partition-aware joins to populate without an API break. Verified by a dedicated test.

## API

```ts
const rows = invoices.query()
  .where('status', '==', 'open')
  .join<'client', Client>('clientId', { as: 'client' })
  .toArray()
// → [{ id, amount, status, client: { id, name, ... } | null }, ...]
```

## Architecture notes

- New structural bridge on `Collection`: `joinResolver` (sibling of the existing `refEnforcer`), implemented by `Compartment` via the new `resolveRef` + `resolveSource` methods. Same decoupling pattern — Collection stays ignorant of Compartment, Compartment stays ignorant of the join executor.
- `Collection.querySourceForJoin()` returns a minimal `{ snapshot, lookupById }` view backed by the in-memory cache. Lazy-mode collections throw because their bounded cache could silently miss records.
- Joins run **after** `where` / `orderBy` / `limit` / `offset` — the left set is filtered and paginated first, then joined. Sorting *by* a joined field is out of scope for v1; post-sort in userland.
- `count()` deliberately ignores joins because joins are projection-only in v0.6 (they attach an aliased field, never filter). Running them just to discard aliases would be wasteful, and in strict mode it could throw on a call whose intent is purely to count.
