## Summary

- Admins can now grant and revoke other admins, removing the v0.4 bottleneck where every new admin had to go through the single `owner` principal.
- Two guardrails: `PrivilegeEscalationError` on widening grants (structurally trivial today but wired in for v0.6+ deputy-admin work), and cascade-on-revoke via the new `RevokeOptions.cascade` field (`'strict'` default, `'warn'` opt-in for diagnostic dry runs).
- Owner is still unrevocable. Single key-rotation pass at the end of a cascade — O(records in affected collections), not O(records × cascade depth).

Closes #62.

## Why these guardrails

**Privilege escalation.** Under v0.5 the admin model gives every admin the full caller DEK set on grant, so \`PrivilegeEscalationError\` is structurally unreachable in typical flows. The check and error class exist so that future per-collection admin scoping (deputy-admin work in v0.6+) cannot accidentally bypass the subset rule. The guard is already there; future code just has to preserve it.

**Cascade on revoke.** Without cascade, revoking \`admin-A\` would leave every admin \`admin-A\` granted in place as orphans with their own chain of descendants — a delegation graph with no owner. The \`'strict'\` default keeps the delegation tree consistent. The \`'warn'\` mode exists for operators who want to see the subtree before cleaning it up manually.

The walk uses the \`granted_by\` field already recorded on every keyring file as the parent pointer — **no on-disk format change**.

## Cost analysis

A 3-level admin cascade (\`owner → A → B → C\`) revoking \`A\`:
- Delete 3 keyring files: O(3 adapter deletes)
- Collect affected collections into a union set: O(collections) memory
- Single rotation pass over the union: O(records in affected collections)

Without the union optimization, a naive per-descendant rotation would be O(records × 3). With the union, it's O(records) regardless of depth. This matches the issue's \"Rotates DEKs once\" comment.

## Breaking changes

**None for existing code.** The semantic change is \"admin granting admin no longer throws\" — code that depended on the throw path is broken, but the only legitimate caller of such code was the v0.4 access-control tests, which have been updated in this PR.

The old behavior (admin cannot grant admin) is no longer recoverable, which is the intended outcome of #62. If a downstream consumer wants to reject admin-grants-admin in their application, they should do so at the application layer before calling \`grant()\`.

## What's NOT in this PR

- **Per-collection admin scoping (\"deputy admin\").** The \`PrivilegeEscalationError\` check is wired in for this, but the actual per-collection scoping API is tracked under v0.6+ work. Today, admin grants always inherit the full caller DEK set.
- **Delegation-tree introspection.** There's no \`listDelegationTree()\` or \`whoGrantedWhom()\` API. The \`granted_by\` field is introspectable via \`listUsers()\` already; a structured tree view is a separate v0.6+ nice-to-have.
- **Grant-time approval flow.** No \"admin-2 must approve admin-1's promotion\" workflow. That's a v0.7+ identity-and-sessions concern.

## Insight for reviewers

- **The cascade walk uses \`granted_by\` directly instead of a derived \"delegation tree\" structure.** The keyring file was already storing the parent pointer in v0.4 (for audit trail attribution), so the tree is reconstructable by walking all keyring files once. No new on-disk field, no new invariant to maintain.
- **The visited set in \`findAdminDescendants\` handles a subtle case:** an admin revoked and later re-granted can end up with \`granted_by\` pointing to someone who is themselves a descendant in the old tree — i.e., a cycle. The visited set terminates the walk cleanly, and the cycle is structurally benign because any descendant already revoked earlier in the walk is already gone.
- **\"Single rotation pass at the end\" matters more than it sounds.** A naive implementation would call \`rotateKeys\` once per revoked descendant, which would be O(records × depth). The union accumulator turns it into O(records), which is the same cost as a single non-cascading revoke. Cascade depth becomes free beyond the walk itself.
