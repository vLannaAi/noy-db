## Summary

Fourth v0.4 feature: **foreign-key references via \`ref()\`**. Collections declare references as metadata; the compartment enforces them on every put/delete and exposes a \`checkIntegrity()\` walk for orphan reporting.

## API

\`\`\`ts
import { ref } from '@noy-db/core'

const company = await db.openCompartment('demo-co')
const clients = company.collection<Client>('clients')

const invoices = company.collection<Invoice>('invoices', {
  refs: {
    clientId: ref('clients'),               // strict (default)
    categoryId: ref('categories', 'warn'),
    parentId: ref('invoices', 'cascade'),   // self-reference OK
  },
})

await invoices.put('inv-1', { id: 'inv-1', clientId: 'c-1', /* ... */ })
//   → throws RefIntegrityError if 'c-1' doesn't exist in 'clients'

await clients.delete('c-1')
//   → throws RefIntegrityError if any strict-referencing invoices still exist

const { violations } = await company.checkIntegrity()
//   → reports every broken reference, with mode + collection + field metadata
\`\`\`

## Three modes

| Mode | put() with missing target | delete() of target with refs |
|---|---|---|
| **strict** (default) | Throws \`RefIntegrityError\` | Throws \`RefIntegrityError\` |
| **warn** | Allowed | Allowed |
| **cascade** | Allowed | Cascades (deletes referencing records) |

\`checkIntegrity()\` reports orphans regardless of mode — useful for warn-mode collections that allow imports from messy sources.

## Design highlights

1. **Refs are metadata, not schema magic.** \`ref()\` returns a plain \`{ target, mode }\` descriptor. No interaction with the Standard Schema validator — keeps validation and reference enforcement as orthogonal concerns. The two compose naturally because they run at different points: schema first (shape correctness), refs second (relational correctness), then encryption.

2. **Atomicity at the put boundary.** Strict ref enforcement runs AFTER schema validation but BEFORE any write. A failed strict put leaves **nothing** on disk, in history, or in the ledger — verified by an explicit test. Same for delete: a strict block on delete leaves the target intact and writes no orphan ledger entries.

3. **Cycle-safe cascade** via a \`cascadeInProgress\` Set. Two collections that cascade-delete each other terminate after one round trip instead of recursing forever. Tested.

4. **Cross-compartment refs rejected at construction.** \`ref('other/clients')\` throws \`RefScopeError\` immediately — they need an auth story (multi-keyring reads) that v0.4 doesn't ship. Tracked for v0.5.

5. **Compartment owns the registry.** A single \`RefRegistry\` per compartment lets cascade traverse multiple collections with one shared cycle breaker. Collections register their refs at construction time; the Compartment's \`enforceRefsOnPut\`/\`enforceRefsOnDelete\` are called from \`Collection.put\`/\`Collection.delete\` via a structural-typed \`refEnforcer\` back-reference (avoids the circular import on \`Compartment\`).
