## Summary

New `.noydb` binary container format that wraps `compartment.dump()` in a thin minimum-disclosure wrapper, plus file-adapter helpers for path-based round-trips. Closes the v0.6 milestone.

```ts
import { writeNoydbBundle, readNoydbBundle } from '@noy-db/core'
import { saveBundle, loadBundle } from '@noy-db/file'

// Core primitives
const bytes = await writeNoydbBundle(company)
const { header, dumpJson } = await readNoydbBundle(bytes)

// File adapter helpers
const handle = await company.getBundleHandle()
await saveBundle(\`./bundles/\${handle}.noydb\`, company)
const result = await loadBundle(\`./bundles/\${handle}.noydb\`)
```

## Why a binary container at all?

`compartment.dump()` already produces a JSON string with encrypted records inside. The wrap is what makes the file safe to drop into cloud storage (Drive, Dropbox, iCloud) without leaking the compartment name and exporter identity through the cloud's metadata API. The minimum-disclosure header is the only thing visible without downloading and decompressing the body.

## Format byte layout

```
+--------+--------+--------+--------+
|  N=78  |  D=68  |  B=66  |  1=49  |  Magic 'NDB1'
+--------+--------+--------+--------+
| flags  | compr  |  header_length (uint32 BE)            |
+--------+--------+--------+--------+--------+--------+--------+
| header_length bytes of UTF-8 JSON header                       ...
+--------+--------+
| compressed body bytes                                            ...
```

10-byte fixed prefix, variable-length JSON header, then the compressed body.

## Minimum-disclosure header

**Only four allowed keys**, validated at parse time:

\`\`\`json
{
  \"formatVersion\": 1,
  \"handle\": \"01HYABCDEFGHJKMNPQRSTVWXYZ\",
  \"bodyBytes\": 41234567,
  \"bodySha256\": \"abc123...\"
}
\`\`\`

**Forbidden** (every key produces a parse error): \`compartment\`, \`_compartment\`, \`exporter\`, \`_exported_by\`, \`timestamp\`, \`_exported_at\`, KDF params, salt fields, any key starting with \`_\`. The validator allowlist is closed — forward-compat extension keys require a format version bump.

## Compression

- **Brotli** when supported (Node 22+, Chrome 124+, Firefox 122+) — 30-50% smaller for JSON
- **gzip** fallback (always available in Node 18+)
- **none** option for round-trip testing

The writer feature-detects brotli at runtime and falls back automatically. The format byte at offset 5 encodes the algorithm so readers handle either transparently.

## Stable opaque handles via ULID

\`compartment.getBundleHandle()\` generates and persists a 26-char Crockford base32 ULID on first call to a reserved \`_meta/handle\` envelope, returns the same handle on subsequent calls. The persistence path bypasses the AES-GCM crypto layer the same way \`_keyring\` already does. Survives process restarts.

The ULID is the only identifier in the bundle header — opaque, doesn't leak compartment names, planned primary key for v0.11 cloud bundle adapters.

## Integrity verification

\`bodyBytes\` and \`bodySha256\` describe the **compressed** body bytes (not the decompressed dump). This lets \`readNoydbBundleHeader()\` verify integrity without decompressing — useful for fast cloud-side validation. Tampering produces \`BundleIntegrityError\`. Length check fires before SHA check (cheaper, more actionable error). Both surface as \`BundleIntegrityError\`, distinct from format errors so consumers can pattern-match the corruption case.

## Why split read from load?

\`readNoydbBundle\` returns the unwrapped dump JSON string, NOT a restored Compartment. Restoring requires a separate \`compartment.load(json, passphrase)\` call. Splitting the layers keeps the bundle module purely a format layer and lets the same code feed format inspectors that never decrypt anything.
