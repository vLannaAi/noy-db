## Summary

Removes the racing \`changesets\` job from \`.github/workflows/release.yml\` and documents the manual release procedure in \`CONTRIBUTING.md\`.

## Why

During the v0.3.0 release, the workflow had **two jobs racing against each other** on push to main:

| Flow | Trigger | Outcome on v0.3.0 |
|---|---|---|
| **A** Changesets-driven | \`push: branches: [main]\` | ❌ Failed with misleading \"E404 PUT\" errors |
| **B** Release-event-driven | \`release: types: [published]\` | ✅ Published all 8 packages cleanly with provenance |

Flow A's failure was caused by a brittle interaction between \`changesets/action@v1\` and the previously-buggy \`linked: [[\"@noy-db/*\"]]\` config (since fixed in #30). The deeper issue is structural: when both jobs fire on the same release, they race on \`pnpm publish\`, and one of them is going to lose.

Manual version bumps + a hand-created GitHub Release is simpler and deterministic, and is what we ended up doing for v0.3.0 anyway.

## Changes

- **\`.github/workflows/release.yml\`** — removed the \`changesets\` job and the \`push: branches: [main]\` / \`workflow_dispatch\` triggers. Workflow now only fires on \`release: published\`. Header comment captures the rationale.
- **\`CONTRIBUTING.md\`** — added a new section **\"7. Releasing to npm\"** with the canonical 7-step manual release procedure (release branch → version bump → CHANGELOG → ROADMAP → PR → GitHub Release → verify). Includes the \`registry.npmjs.com\` vs \`.org\` CDN-cache gotcha we hit on v0.3.0.
