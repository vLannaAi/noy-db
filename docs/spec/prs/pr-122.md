## Summary

Adds `Query.groupBy(field)` — the second leg of v0.6 aggregations. Partitions matching records into buckets keyed by a field, terminating with `.aggregate(spec)` to compute per-bucket reducers. Reuses the reducer protocol from #97 directly — no duplicated API.

```ts
const byClient = invoices.query()
  .where('status', '==', 'open')
  .groupBy('clientId')
  .aggregate({ total: sum('amount'), n: count() })
  .run()
// → [ { clientId: 'c1', total: 5250, n: 3 }, … ]
```

## Design choices

**Bucket ordering:** first-seen insertion order. JS `Map` preserves it natively, so no parallel order array is needed. Consumers who want a specific ordering should `.sort()` downstream.

**Cardinality caps:**
- **10_000 groups** — one-shot warning (`GROUPBY_WARN_CARDINALITY`), deduped per-field so reactive dashboards re-executing the same query don't spam the console
- **100_000 groups** — hard `GroupCardinalityError` (`GROUPBY_MAX_CARDINALITY`)

The cardinality check runs **inside the partition loop** so the 100_001st bucket throws at the moment it would be created, not after the full partition materializes — the consumer's OOM pressure is bounded by the cap, not by the query's match count.

**Null vs undefined keys** are deliberately separate buckets. `Map` distinguishes them natively and merging would hide a real distinction between "field explicitly set to null" and "field not set at all". Consumers who want them merged should coalesce upstream with `.filter()`.

**Live mode** reuses the same reactive primitive as `.aggregate().live()` via a new `buildLiveAggregation` helper exported from `aggregate.ts` — avoids reaching into the private `LiveAggregationImpl` class directly. Naive full re-run on every source change; per-bucket incremental delta maintenance is a future optimization that the reducer protocol already admits.

**Joins skipped** in grouped pipelines — same rationale as `.count()` and `.aggregate()` from #97. Joined fields in v0.6 are projection-only, so running a join inside a grouping pipeline would be wasteful and could trigger `DanglingReferenceError` in strict mode.

**Type-level stable-key narrowing (v0.8 #85 prep):** v0.6 types the group key as `unknown`. When `dictKey` lands in v0.8, a `groupBy<DictField>()` overload will narrow the group key to the stable dictionary key rather than the resolved locale label — preventing the silent bug where grouping by a localized label produces different buckets per reader. The overload layers on top without an API break.
