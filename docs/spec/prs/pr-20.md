## Summary

Adds the `@noy-db/pinia` package and a Vue 3 + Vite + Pinia reference playground demonstrating the v0.3 one-line greenfield path.

```ts
// stores/invoices.ts
import { defineNoydbStore } from '@noy-db/pinia';

export const useInvoices = defineNoydbStore<Invoice>('invoices', {
  compartment: 'C101',
});
```

That's all the user writes. Behind the scenes the store wires up:

- An encrypted NOYDB collection
- Reactive `items` / `count` / `byId`
- `add` / `update` / `remove` / `refresh` mutations
- The chainable `query()` DSL from #19 (this PR depends on it)
- A `$ready` Promise for hydration handshakes
- Schema validation via any Standard-Schema-compatible parser

## Notes for reviewers

### Variance

\`Compartment.collection<T>()\` caches \`Collection<unknown>\` and casts on retrieval. This works because the v0.3 query DSL keeps \`Collection<T>\` covariant in T (see the variance discussion in the query DSL PR #19).

### Why \`shallowRef\` for \`items\`?

Per-record proxying via deep \`ref()\` is expensive at 50K records and unnecessary — when records change we replace the whole array, which is the trigger Vue actually needs.

### \`add()\` / \`update()\` re-list after every mutation

An optimization opportunity: splice into \`items.value\` directly rather than re-listing. Deferred because re-listing is correct, simple, and fast at the v0.2 target scale (1K–50K records). Will revisit when #15 (lazy hydration + LRU) lands.

### Playground design choices

Used the \`@noy-db/browser\` adapter (IndexedDB) so reviewers can open DevTools → Application → IndexedDB and see the ciphertext directly. The README explains exactly what to look for. Vite target was bumped to es2022 because main.ts uses top-level await to construct the Noydb instance before mounting Vue.
