## What ships

### Bin 1: `create-noy-db` (the wizard)

Invoked by \`npm create noy-db@latest\`, \`pnpm create noy-db\`, \`yarn create noy-db\`, and \`bun create noy-db\`. Asks at most 3 questions and generates a fully wired Nuxt 4 + Pinia + encrypted-store starter:

\`\`\`bash
npm create noy-db@latest my-app                       # interactive
npm create noy-db@latest my-app --yes                 # all defaults
npm create noy-db@latest my-app --yes --adapter file  # pick a different adapter
\`\`\`

### Bin 2: \`noy-db\` (the tool)

Ongoing commands for existing projects, invoked via \`pnpm exec noy-db <cmd>\` or \`npx noy-db <cmd>\`:

- \`noy-db add <collection>\` — scaffolds \`app/stores/<name>.ts\` and \`app/pages/<name>.vue\`. Refuses to overwrite; atomic (either both files land or neither does).
- \`noy-db verify\` — end-to-end crypto round-trip check against an in-memory adapter. Validates that \`@noy-db/core\`, \`@noy-db/memory\`, and the local Node version all agree on Web Crypto. Exits non-zero on any divergence.

## Scope decision vs the original issues

Original #9 asked for a \`nuxi noydb <command>\` extension inside \`@noy-db/nuxt\`. I shipped a standalone \`noy-db\` bin in this package instead. **Why:**

1. Nuxt 4's CLI extension API has changed across multiple releases. Coupling to it would trap us in whichever version is current.
2. A standalone bin works in non-Nuxt projects too, is portable across package managers, and is trivially testable without a real terminal.
3. \`@noy-db/nuxt\` doesn't need to change — no race-prone version bumps to an already-published package.

Same functional value, smaller blast radius. Documented in the PR diff + the package's README "Deferred to a follow-up" section.
