## Summary

- Replaces the v0.4 owner-only `Compartment.export()` with `exportStream()` + `exportJSON()` — both ACL-scoped, both surface schema/refs metadata, both carry the plaintext-on-disk warning block.
- `exportStream()` is the primitive: an `AsyncIterableIterator<ExportChunk>` yielding per-collection chunks by default, opt-in `granularity: 'record'` for arbitrarily large collections, opt-in `withLedgerHead: true` for chain-anchored exports.
- `exportJSON()` is the universal default helper: `Promise<string>` with a stable on-disk shape. Browser-safe — core has zero `node:` imports, so the consumer chooses any sink (`fs.writeFile`, `Blob` download, `fetch` upload, IndexedDB).

Closes #72.

## Why a string instead of a file path

Core runs in browsers, Node, Bun, Deno, and edge runtimes. Accepting a path would force a `node:fs` import (breaks browsers) or a runtime dynamic import (doesn't tree-shake, inflates bundles). Returning a string lets the consumer choose any sink and forces the destination decision to be explicit at the call site — which is also better for the security warning, since there's no library function quietly writing plaintext somewhere.

## Why ACL-scoped instead of owner-only

The old `export()` threw `PermissionDeniedError` for non-owners. That's the wrong shape: an operator with `{ invoices: 'rw' }` permissions should be able to export the invoices they can already read, just not the collections they can't. The new behavior matches `Collection.list()` — silent skip on inaccessible collections, no error. Operator with one grant on a five-collection compartment exports one collection, not zero.

## Composition

Once `queryAcross()` lands (next #63 issue in v0.5), fanning the export across every compartment the caller can unlock is `queryAcross(ids, c => c.exportStream())` — no new primitive needed. That's part of why this method belongs in core: it's the single decrypt+ACL+metadata path that every export-format package will build on.

## Breaking changes

- `Compartment.export()` removed. One-line migration: `comp.export()` → `comp.exportJSON()`.
- Non-owner export no longer throws — returns only the collections the caller can read instead.
- The \"operator cannot export\" assertion in `access-control.test.ts` was updated to assert the new ACL-scoped behavior.

## What's NOT in this PR

- CSV / XML / xlsx serializers — separate `@noy-db/decrypt-*` packages, milestoned for v0.6+.
- Importer (round-tripping plaintext back into a compartment) — separate v0.6+ feature with its own auth + integrity questions.
- True per-record adapter streaming — gated on a v0.6 adapter-contract extension. The current \"streaming\" is on the output side (no giant accumulator); the underlying adapter call is still a single \`loadAll\`.

## Insight for reviewers

- The decision to put \`schema\` on every chunk (instead of in a header) turned out to be load-bearing for the per-record granularity case: per-record consumers don't have to thread state across yields. The \"wasteful repetition\" is consistency that makes consumer code one branch shorter. Same argument applies to \`ledgerHead\` — per-chunk slot is forward-compatible with v0.10 partition work where the value will genuinely differ per chunk, with zero cost today.
- \`exportJSON\` always sets \`schema: null\` in the JSON output (Standard Schema validators are not JSON-serializable). Format-package serializers that need the schema use \`exportStream()\` directly and read \`chunk.schema\` (the live validator object). \`exportJSON\` is a \"dump everything\" convenience; format-aware exporters use the primitive.
