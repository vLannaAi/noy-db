## Summary

Adds opt-in lazy collection hydration with an LRU cache, so collections larger than RAM can be served on-demand. The default \`prefetch: true\` keeps **v0.2 behavior unchanged** — eager mode still loads everything into memory on first access.

\`\`\`ts
const invoices = compartment.collection<Invoice>('invoices', {
  prefetch: false,
  cache: { maxRecords: 1000, maxBytes: '50MB' },
});

await invoices.get('inv-001');                              // adapter.get on miss; populates LRU
await invoices.put('inv-002', record);                      // write-through, inserts in LRU
for await (const r of invoices.scan({ pageSize: 500 })) {}  // streams via #14, does NOT pollute the LRU

// Throws in lazy mode (clear redirect to scan):
// await invoices.list()
// invoices.query()

// Diagnostics:
invoices.cacheStats() // { hits, misses, evictions, size, bytes, lazy }
\`\`\`

## What's deferred (and why)

Two items from the spec did NOT ship in this PR:

1. **Dirty record pinning** — sync-pending records "pinned" against eviction. The sync engine doesn't currently expose per-collection dirty state in a way the LRU can consult. Apps using lazy mode + sync should call \`scan()\` (or hydrate the dirty set manually) before pushing. Pinning will land alongside the v0.4 sync refresh.

2. **Indexes + lazy mode integration** — the spec mentioned re-hydrating evicted records on indexed query hits. v0.3 takes the conservative path: \`{ prefetch: false, indexes: [...] }\` is **rejected at construction time**. The error message points at the v0.4 follow-up. This avoids silent correctness bugs (a query returning stale results because some matching records have been evicted from the cache).

Both are documented in the commit body and the changeset.

## Trade-offs

### Why two clean modes (eager OR lazy) instead of "eager + LRU cap"?

I considered a hybrid mode where the collection eager-loads and then LRU-evicts cold records as new ones arrive. Rejected: it creates silent footguns. \`list()\` would return only the cached subset (semantically wrong); \`query()\` would silently miss evicted records; \`count()\` would be ambiguous. Two clean modes (\`prefetch: true\` for full memory, \`prefetch: false\` for lazy) keeps semantics unambiguous.

### Why does \`scan()\` skip the LRU?

If \`scan()\` populated the LRU, streaming a 100K-record collection would turn the LRU into a write-once buffer that immediately evicts everything (because each new page evicts the previous one). The LRU is for **random-access workloads**, not sequential ones. \`scan()\` already has its own pagination via #14; it doesn't need the LRU.

### Why does \`put()\` need to fetch the previous record from the adapter in lazy mode?

History snapshots need the previous version's record. In eager mode the previous record is always in the cache (because everything is hydrated). In lazy mode it might have been evicted or never loaded — so \`put()\` does an \`adapter.get()\` to fetch it before writing the history snapshot. The cost is one extra adapter call per update; acceptable because (a) most updates are on hot records that ARE in the cache, and (b) history can be disabled if it's a problem.

## Notes for reviewers

### The off-by-one keyring read gotcha (test infra)

My first attempt at the integration tests had two failing assertions because \`openCompartment()\` triggers a keyring read via \`adapter.get()\`, which polluted the get-call counter. The fix was to have the test helper open the compartment AND reset the counter, then return the opened compartment to test bodies. Tests reference \`comp\` directly instead of re-opening. Worth knowing for any future test that counts adapter calls.

### One transient flake during full-suite run

While iterating, I saw a one-time failure of the existing index speedup test (#22 from the indexes PR) — the 3× gate dropped to ~2.4× under heavy parallel load. Re-running the suite immediately gave 269/269 passing. Not a regression; the test is just on the edge of CI stability under maximum parallel load. If it flakes again on the CI runner, the gate could be lowered to 2.5× without losing the "indexes are measurably faster" claim.
