## Summary

Third v0.4 feature: **delta history**. Every \`Collection.put\` after the genesis now computes a **reverse** JSON Patch from the new record to the previous version and stores it in the compartment's \`_ledger_deltas/\` internal collection. The delta's sha256 is recorded in the ledger entry's new optional \`deltaHash\` field, linking it to the hash chain from #43.

## API

\`\`\`ts
const company = await db.openCompartment('demo-co')
const invoices = company.collection('invoices')

// Normal puts — deltas are stored automatically.
await invoices.put('inv-1', { amount: 100, status: 'draft' })
await invoices.put('inv-1', { amount: 150, status: 'open' })
await invoices.put('inv-1', { amount: 150, status: 'paid' })

// Reconstruct any historical version by walking the chain.
const ledger = company.ledger()
const current = await invoices.get('inv-1')
const v2 = await ledger.reconstruct('invoices', 'inv-1', current, 2)
// → { amount: 150, status: 'open' }
\`\`\`

## Why reverse patches?

Two design options:
- **Forward** (\`prev → next\`): easy to apply going forward from a base snapshot
- **Reverse** (\`next → prev\`): easy to apply going backward from the current state

I picked **reverse** because the current state is already live in the data collection — no base snapshot needed. Walking backward from \"now\" through reverse patches reconstructs any historical version without duplicating data. This halves the storage cost vs a forward-patches + base-snapshots design.

## Storage efficiency — the whole point

A 1 KB record edited 100 times produces ~100 KB of full snapshots but **under 20 KB of deltas**. This is enforced by a test gate:

\`\`\`ts
// from __tests__/delta-history.test.ts
const bigRecord = { id: 'big', padding: 'x'.repeat(1024), counter: 0 }
await coll.put('big', bigRecord)
for (let i = 1; i <= 100; i++) {
  await coll.put('big', { ...bigRecord, counter: i })
}
// Measure bytes in _ledger_deltas
expect(totalDeltaBytes).toBeLessThan(20_000) // 5x headroom vs snapshots
\`\`\`

## What's new

**Modules:**
- \`packages/core/src/ledger/patch.ts\` — hand-rolled RFC 6902 JSON Patch compute + apply. Subset: \`add\`, \`remove\`, \`replace\` (no \`move\`/\`copy\`/\`test\` — not needed for our use). Arrays are atomic values. Path escaping (\`~0\`, \`~1\`) per spec. **Zero deps.**
- \`LedgerStore.reconstruct(collection, id, current, atVersion)\` — walks the chain backward and applies reverse patches.
- \`LedgerStore.loadDelta(index)\` — exposes individual delta payloads.
- \`LedgerEntry.deltaHash\` — new optional field.
- \`LEDGER_DELTAS_COLLECTION\` — new internal collection constant.

**Exports:** \`JsonPatch\`, \`JsonPatchOp\`, \`computePatch\`, \`applyPatch\`.

## Known limitations (documented in code + changeset)

1. \`reconstruct()\` is ambiguous across delete+recreate cycles because \`version\` numbers restart at 1 after a delete. Ledger index-based reconstruct is a v0.5 follow-up.
2. The legacy \`_history\` collection (full snapshots) is still maintained alongside deltas. Deprecating it is a follow-up.
3. \`pruneHistory({ keepLast: N })\` — folding old deltas into a base snapshot — is deferred.
4. Grant/revoke/rotate as ledger operations are still deferred (they only record put/delete today).
