## Summary

New terminal `.live()` on `Query<T>` returns a `LiveQuery<T>` ‚Äî a framework-agnostic reactive primitive with `value` / `error` fields and a `subscribe(cb)` notification channel.

- **Non-joined queries:** convenience over the existing `.subscribe()` callback shape.
- **Joined queries:** additionally subscribes to every join target's change stream so right-side mutations re-fire the live query and re-evaluate every dependent left row. Right-side targets are deduped by collection name so a chain joining the same target twice only subscribes once.

LiveQuery preserves the previous successful value when a re-run throws and stores the error in `live.error` instead of propagating the throw out of the source's change handler (which would tear down the upstream emitter). Consumers check `live.error` after each notification and render an error state.

## Ref-mode behavior on right-side disappearance

Matches the eager `.toArray()` contract from #73:

- **strict** ‚Üí re-run throws `DanglingReferenceError`, caught and stored in `live.error`, listeners notified
- **warn** ‚Üí joined value flips to `null`, existing one-shot warn dedup keeps repeats from spamming the console
- **cascade** ‚Üí no special handling ‚Äî v0.4 cascade-delete propagates the right-side delete and the live query naturally re-fires with orphans gone

## Implementation

- New `query/live.ts`: `LiveQuery` interface, `LiveUpstream` interface, `buildLiveQuery` factory, `LiveQueryImpl` class with error-isolated upstream subscriptions, listener errors, idempotent `stop()`, no-op subscribe after stop
- `JoinableSource.subscribe?`: optional method on the join-source interface (#73's `join.ts`), populated by Collection's `querySourceForJoin()` so right-side mutations propagate
- `Query.live()`: collects upstreams (left `source.subscribe` + each unique join target's `source.subscribe` via `joinContext.resolveSource`), builds a `LiveQuery` whose recompute is bound to `this.toArray()`
