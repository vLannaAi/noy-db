## Summary

Adds the optional \`listPage()\` adapter capability and the \`Collection.scan()\` async iterator so collections larger than RAM can be processed page-by-page. The 6-method core adapter contract is unchanged — \`listPage\` is purely additive.

\`\`\`ts
// Stream every record without loading the whole collection into memory
for await (const record of invoices.scan({ pageSize: 500 })) {
  await processOne(record);
}

// Or fetch a single page directly
const page = await invoices.listPage({ limit: 100 });
\`\`\`

**All five built-in adapters implement listPage natively.** Adapters that don't get a synthetic fallback over \`list() + get()\` with a one-time \`console.warn\`.

## Notes for reviewers

### Why \`items: Array<{ id, envelope }>\` instead of just envelopes

The page result includes both the id and the envelope. This avoids an N+1 round-trip pattern: without the id, callers would need a separate call to map envelopes back to ids for any post-processing. Returning them paired matches what every consumer actually needs.

### Why no Buffer in dynamo

The dynamo package has zero non-AWS dependencies and we want to keep it that way. The cursor encoder uses \`btoa(unescape(encodeURIComponent(input)))\` instead of \`Buffer.from(input).toString('base64')\` so we don't need to add \`@types/node\`. This works in both Node 20+ and any modern browser.

### Why s3 client injection cast through \`as AwsS3Client\`

The new \`S3ClientLike\` interface only declares \`send(command): Promise<unknown>\`, but the existing s3 methods rely on the typed responses from \`@aws-sdk/client-s3\`. The cast through \`AwsS3Client\` is structurally safe: every command's response shape is documented in the AWS SDK, and the test mocks return matching shapes. A stricter typed approach would require duplicating the entire AWS SDK response type tree — not worth the maintenance burden for a test injection point.

### \`listPage\` returns full envelopes, not just ids — by design

This is the load-bearing API choice. Returning ids would force callers into an N+1 \`get()\` per page, which defeats the purpose of pagination. Every adapter pays a tiny cost to inline the bodies into the page result, but this is dominated by the per-record fetch cost regardless.

### What's NOT in this PR

- 100K-record memory benchmark (see acceptance criteria above)
- The lazy hydration + LRU eviction work (#15) — that's the natural follow-up
- Browser IndexedDB pagination tests — IndexedDB needs happy-dom setup which is more invasive than I wanted in this PR. The localStorage backend's listPage IS exercised by the existing browser conformance tests via \`list()\`.
