## Summary

Adds in-memory hash indexes to \`@noy-db/core\` so the query DSL from #19 can short-circuit \`==\` and \`in\` lookups instead of always doing a linear scan. Indexes are declared per-collection and maintained automatically.

\`\`\`ts
const invoices = compartment.collection<Invoice>('invoices', {
  indexes: ['status', 'client'],
});

invoices.query()
  .where('status', '==', 'open')   // ← uses the index
  .where('amount', '>', 1000)      // ← unindexed, falls back to scan
  .toArray();
\`\`\`

## What's deferred (and why)

The issue spec mentions persisting indexes as encrypted blobs alongside records. **v0.3 ships in-memory indexes only.** Reason: at the v0.3 target scale of 1K–50K records, hydrate already loops over every record and decrypts it; building the index during that loop adds essentially zero cost. Persistence would only matter at >100K records where hydrate itself becomes a bottleneck — and at that scale you'd want a real database, not NOYDB.

The escape hatch: \`CollectionIndexes\` is the only file that touches index data, so adding encrypted blob persistence later is purely additive. I'd open a follow-up issue for it once we have benchmarks showing hydrate is the bottleneck for any real consumer.

Composite/sorted/unique indexes follow the same logic — they're additive variants of \`IndexDef\` and can land in v0.4 without breaking the v0.3 API.

## Notes for reviewers

### The performance gotcha

My first attempt at the executor only saw a **2.4× speedup** because it re-evaluated the index-driving clause against the candidates after the index lookup. The fix was to have \`candidateRecords()\` return BOTH the candidates AND the *remaining* clauses to evaluate — for a single-clause indexed query, the remaining list is empty and the executor skips per-record predicate evaluation entirely. That brought the ratio up to 4–6× consistently. Watch out for this when adding more index types — the planner contract is \"index hits exclude their driving clause from the remainder list\".

### Variance

\`IndexDef\` is currently \`type IndexDef = string\` for v0.3. Future variants (\`{ fields, unique }\`, etc.) land as discriminated unions of this alias without breaking existing declarations. The migration is purely additive.

### \`getIndexes()\` and \`lookupById()\` are optional on \`QuerySource\`

The query builder works against any \`QuerySource\` — in-memory arrays, the Collection's cache, custom test sources. Only sources that implement BOTH \`getIndexes()\` and \`lookupById()\` get the indexed fast path. Sources without them fall back to a full scan. This keeps the indexes feature opt-in without breaking the framework-agnostic Query API.

### Adapter spy test (#20) — keyring puts

The keyring file (\`_keyring/{userId}.json\`) is stored via \`adapter.put()\` but it's NOT a NOYDB record envelope — it's a JSON document containing AES-KW-wrapped DEKs. So its \`_iv\` field is empty, which would have failed the spy assertion. The test now filters \`_putCalls\` to only the target collection (\`'invoices'\`), excluding keyring writes. The keyring's contents are still wrapped DEKs (encrypted) — only the envelope wrapper is plain JSON.
