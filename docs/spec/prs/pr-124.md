## Summary

`ScanBuilder` now has a chainable `.join(field, { as })` method that resolves a `ref()`-declared foreign key per record as the scan stream flows, attaching the right-side record (or null) under the alias. Reuses the eager join's `JoinContext` / `JoinableSource` / `DanglingReferenceError` infrastructure from #73 ‚Äî no duplicated machinery ‚Äî and matches its ref-mode semantics exactly.

```ts
// Streaming joined iteration
for await (const inv of invoices.scan().join('clientId', { as: 'client' })) {
  await processInvoice(inv) // inv.client is attached
}

// Streaming joined aggregation
const { total } = await invoices.scan()
  .where('status', '==', 'open')
  .join('clientId', { as: 'client' })
  .aggregate({ total: sum('amount') })
```

## Key difference from eager `Query.join()` (#73)

**The LEFT side streams** page-by-page from the adapter and is never materialized. Memory ceiling on the left is O(pageSize), not O(rowCount) ‚Äî this is what makes streaming joins suitable for collections that exceed eager join's 50_000-row ceiling.

**The right side is held in memory** for the duration of the iteration in both strategies. True left-and-right streaming joins (sort-merge planner) are out of scope for v0.6.

## Right-side strategy (auto-selected per leg)

| Strategy | Trigger | Per-row cost | Upfront cost |
|---|---|---|---|
| **Indexed** | source has `lookupById` | O(1) | none |
| **Hash** | source has only `snapshot()` | O(1) | O(N) once |

The indexed path is the common case for Collection right sides ‚Äî Collection's `querySourceForJoin()` returns a JoinableSource with `lookupById` backed by the in-memory cache. The hash path is the fallback for custom synthetic sources.

## Ref-mode semantics ‚Äî identical to eager join

| Mode | Dangling ref behavior |
|---|---|
| `strict` | Throws `DanglingReferenceError` mid-stream. The throw aborts the async iterator ‚Äî consumers wrap `for await` in try/catch to recover. |
| `warn` | Attaches `null` + one-shot warning per unique dangling pair, deduped per iteration via the same `field‚Üítarget:refId` key. |
| `cascade` | Attaches `null` silently. |

Left records with **null/undefined** FK values attach `null` regardless of mode ‚Äî same "no reference at all" policy as eager join and write-time `enforceRefsOnPut`.

## Multi-FK chaining

Repeated `.join()` calls each resolve an independent ref. Each leg picks its own strategy and ref mode independently.

## Pipeline ordering (clauses then joins)

Joins run **after** clauses in the streaming pipeline, matching the eager `Query.toArray()` ordering. This means `.where()` / `.filter()` can only see un-joined fields ‚Äî filtering on joined aliases requires a post-iteration filter in userland (out of scope for v0.6).

## #87 constraint #1 (partition seam)

Every `JoinLeg` from a streaming join carries `partitionScope: 'all'` plumbed through but never read by v0.6. Same load-bearing seam as eager join.

## Wiring

`Collection.scan()` now passes a `JoinContext` to the `ScanBuilder` it returns ‚Äî same machinery as `Collection.query()` already used for eager joins. `ScanBuilder` constructed via the direct constructor (with a synthetic `ScanPageProvider`) has no `joinContext` and `.join()` throws with an actionable error.

## Out of scope (tracked separately)

- True left-and-right streaming joins (sort-merge planner)
- LRU + lazy probe for non-`lookupById` right sources ‚Äî the current hash-from-snapshot fallback materializes the right side once; LRU only matters when the right side itself is stream-only, which v0.6's adapter API doesn't model
- Filtering on joined fields after `.join()` ‚Äî would need to swap pipeline ordering
- `scan().join().live()` ‚Äî same design problem as `scan().aggregate().live()`
- Streaming join across compartments ‚Äî `queryAcross` continues as the cross-compartment correlation primitive
