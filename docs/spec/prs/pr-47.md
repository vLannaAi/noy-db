## Summary

First v0.4 feature: **schema validation via [Standard Schema v1](https://standardschema.dev)**. Any validator that implements the protocol (Zod, Valibot, ArkType, Effect Schema, \`@standard-schema/spec\`) can now be attached to a \`Collection\` or \`defineNoydbStore\`. Validation runs **before encryption on \`put()\`** and **after decryption on reads**.

## API surface

\`\`\`ts
import { z } from 'zod'

const InvoiceSchema = z.object({
  id: z.string(),
  amount: z.number().positive(),
  status: z.enum(['draft', 'open', 'paid']),
})

export const useInvoices = defineNoydbStore<z.infer<typeof InvoiceSchema>>('invoices', {
  compartment: 'demo-co',
  schema: InvoiceSchema,
})
\`\`\`

New exports from \`@noy-db/core\`:
- Types: \`StandardSchemaV1\`, \`StandardSchemaV1Issue\`, \`StandardSchemaV1SyncResult\`, \`InferOutput\`
- Helpers: \`validateSchemaInput\`, \`validateSchemaOutput\`
- Error: \`SchemaValidationError\` (carries \`issues\` + \`direction: 'input' | 'output'\`)

## Design decisions

1. **Vendored types.** Standard Schema is a types-only protocol. Rather than pull in \`@standard-schema/spec\` purely for types, I vendored the ~60 lines we need directly into \`packages/core/src/schema.ts\`. This keeps \`@noy-db/core\` at **zero runtime dependencies** and gives us freedom to evolve.

2. **Validation at the decrypt boundary, not the store layer.** Centralizing validation inside \`Collection.decryptRecord()\` means every read path (get, list, query, scan, listPage) gets it automatically, for free. The pinia store's old \`options.schema.parse(record)\` call in \`add()\` was removed — duplicate, only covered writes, and couldn't catch drifted stored data.

3. **History reads skip validation.** Historical records predate the current schema by definition — validating them against today's shape would be a false positive on any schema evolution. \`getVersion()\` and \`history()\` pass \`skipValidation: true\` through to \`decryptRecord\`.

4. **\`direction: 'input' | 'output'\` on the thrown error.** Input failures are normal (user sent bad data — UI handles it). Output failures indicate schema drift — an unexpected state that should be loud. Same error class, different semantics.

## Breaking change

\`@noy-db/pinia\`'s \`schema\` option now expects a Standard Schema v1 validator instead of a hand-rolled \`{ parse }\` shim. Zod schemas already implement the protocol; Valibot, ArkType, and Effect Schema too. Consumers who passed a custom \`{ parse }\` object need to wrap it:

\`\`\`ts
// before
schema: { parse: (v) => { /* ... */ } }

// after (minimal Standard Schema v1 shape)
schema: {
  '~standard': {
    version: 1,
    vendor: 'my-validator',
    validate: (v) => ({ value: /* ... */ }), // or { issues: [...] }
  },
}
\`\`\`

See \`packages/pinia/__tests__/defineNoydbStore.test.ts\` for a 10-line example.
