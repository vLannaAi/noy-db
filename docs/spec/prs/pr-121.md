## Summary

Adds the **aggregation leg** of v0.6 query DSL completion. Five reducer factories (`count`, `sum`, `avg`, `min`, `max`) plug into a new `.aggregate(spec)` near-terminal on `Query<T>` that returns an `Aggregation<R>` wrapper with two terminals of its own:

- `.run(): R` — synchronous one-shot reduction
- `.live(): LiveAggregation<R>` — reactive primitive that re-runs the reduction whenever the source notifies of a change

```ts
// Static
const { total, n, meanAmount } = invoices.query()
  .where('status', '==', 'open')
  .aggregate({
    total:      sum('amount'),
    n:          count(),
    meanAmount: avg('amount'),
  })
  .run()

// Reactive
const live = invoices.query()
  .where('status', '==', 'open')
  .aggregate({ total: sum('amount'), n: count() })
  .live()

live.subscribe(() => renderDashboard(live.value))
// ... later
live.stop()
```

## Design choices

**Reducer protocol** is `{ init, step, remove?, finalize }` with separate internal state (`S`) and user-visible result (`R`) type parameters. The optional `remove()` hook is the seam for future O(1) incremental maintenance; v0.6 ships naive full re-run on source change, which is simpler, correct, and lets incremental land as a pure optimization later without an API break.

**Return type inference** flows through the spec via a mapped-type extraction on `Reducer<infer R, unknown>`, so a spec of `{ total: sum('amount'), n: count() }` yields a result of `{ total: number, n: number }`.

**`avg` / `min` / `max` return `number | null`** on empty result sets rather than NaN / ±Infinity — avoiding the foot-gun of poisoning downstream arithmetic in the "no matches" case.

**`LiveAggregation<R>` mirrors `LiveQuery` from #74** in shape and error-isolation contract: `value` / `error` fields readable synchronously, `subscribe(cb)` notification channel, re-run errors preserve the previous successful value and surface through `live.error` instead of propagating out of the source's change handler, `stop()` idempotent, listener errors isolated per-listener.

**Aggregations skip joins in v0.6** — same rationale as `count()`. Joins in v0.6 are projection-only; running them just to throw the aliases away would be wasteful and could trigger `DanglingReferenceError` in strict mode for a call whose intent is purely to aggregate.

## #87 constraint #2 (load-bearing seam)

Every reducer factory accepts an optional `{ seed }` parameter plumbed through the protocol but **unused by the v0.6 executor**. When v0.10 partition-aware aggregation lands, the seed will carry running state across partition boundaries without requiring an API break. Do not remove — that's the whole point of having it now. A dedicated test pins the no-op behavior.
