## Summary

Adds **augment mode** to the \`create @noy-db\` wizard. When invoked inside an existing Nuxt 4 project, the wizard now patches the existing \`nuxt.config.ts\` in place (via magicast AST rewriting) instead of refusing to continue or creating a sub-project.

## What you can now do

\`\`\`bash
cd ~/my-existing-nuxt-app
npm create @noy-db                    # preview the diff and confirm
npm create @noy-db --dry-run          # print the diff, no write
npm create @noy-db --yes              # non-interactive, write immediately
\`\`\`

The wizard auto-detects existing Nuxt 4 projects by looking for **both** \`nuxt.config.{ts,js,mjs}\` AND a \`package.json\` that lists \`nuxt\` in any dependency section. Both alone are ambiguous (could be a stray template file, could be a workspace root with nuxt in docs deps), so requiring both avoids false positives.

## What it does to your config

Via [magicast](https://github.com/unjs/magicast) AST rewriting — not regex, not string patching:

1. Adds \`'@noy-db/nuxt'\` to the \`modules\` array (creates the array if missing)
2. Adds \`noydb: { adapter, pinia: true, devtools: true }\` — **only if the key doesn't already exist**
3. Preserves unrelated config keys, comments, trailing commas, and formatting
4. Shows a colored unified diff before any write
5. Asks for confirmation unless \`--yes\` is passed
6. Prints the install command for the \`@noy-db/*\` packages

Example diff output in the terminal:

\`\`\`diff
-export default defineNuxtConfig({ modules: [] })
+export default defineNuxtConfig({
+  modules: ["@noy-db/nuxt"],
+
+  noydb: {
+    adapter: 'browser',
+    pinia: true,
+    devtools: true
+  }
+})
\`\`\`

## Safe behaviors

| Scenario | Outcome |
|---|---|
| Already augmented (idempotent re-run) | No-op; reports \"already configured\" |
| Pre-existing \`noydb:\` key | Preserved as-is; only \`modules\` is updated if needed |
| \`--dry-run\` | Prints diff, returns without writing |
| Opaque export (\`export default someVar\`) | Rejected with a \"cannot safely patch\" message |
| Non-array \`modules\` | Rejected with the same |
| \`--force-fresh\` | Bypasses detection, creates a new subdirectory even inside an existing Nuxt project (useful for sub-projects and CI tests) |

## Interesting bug caught during dev

First implementation of \`resolveConfigObject\` used \`Array.isArray(proxy.$args)\` to detect the \`defineNuxtConfig({...})\` call shape. That returned false for every input — because magicast's \`$args\` is a **proxy**, not a real array, so \`Array.isArray\` doesn't recognize it. Every test failed with \`unsupported-shape\`. Fixed by indexing \`$args[0]\` directly and checking whether the result is an object. The inline comment on \`resolveConfigObject\` now calls this out explicitly so future-me doesn't repeat it.
